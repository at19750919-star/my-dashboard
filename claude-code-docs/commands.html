檢查點 Checkpointing

追蹤、回溯和摘要 Claude 的編輯和對話以管理會話狀態。Claude Code 自動追蹤 Claude 在工作時所做的檔案編輯，讓您可以快速撤銷變更並回溯到之前的狀態，以防任何事情出現偏差。

檢查點如何運作 How Checkpointing Works

當您與 Claude 合作時，checkpointing 會自動擷取每次編輯前程式碼的狀態。這個安全網讓您可以放心地執行大規模任務，因為您始終可以返回到之前的程式碼狀態。

自動追蹤 Automatic Tracking

Claude Code 追蹤其檔案編輯工具所做的所有變更：

  • 每個使用者提示都會建立一個新的 checkpoint（檢查點）
  • Checkpoints 在會話之間持久存在，因此您可以在恢復的對話中存取它們
  • 在 30 天後自動清理（可設定）

回溯和摘要 Rewind and Summarize

按兩次 Esc（Esc + Esc）或使用 /rewind 命令開啟回溯選單。一個可捲動的列表顯示會話中的每個提示。選擇您想要操作的點，然後選擇一個動作：

  • Restore code and conversation（恢復程式碼和對話）：將程式碼和對話都恢復到該點
  • Restore conversation（恢復對話）：回溯到該訊息，同時保持目前程式碼
  • Restore code（恢復程式碼）：恢復檔案變更，同時保持對話
  • Summarize from here（從此處摘要）：將此點之後的對話壓縮為摘要，釋放 context window（上下文視窗）空間
  • Never mind（算了）：返回訊息列表而不做任何變更

恢復與摘要 Restore vs. Summarize

三個恢復選項恢復狀態：它們撤銷程式碼變更、對話歷史或兩者。「從此處摘要」（Summarize from here）的運作方式不同：

  • 所選訊息之前的訊息保持不變
  • 所選訊息及其後的所有訊息被替換為精簡的 AI 產生摘要
  • 磁碟上的檔案不會改變
  • 原始訊息保存在會話記錄中，因此 Claude 可以在需要時參考詳細資訊

這類似於 /compact，但更有針對性：您不是摘要整個對話，而是保持早期上下文的完整細節，只壓縮佔用空間的部分。您可以輸入選用說明來指導摘要的重點。

注意：摘要將您保持在同一會話中並壓縮上下文。如果您想嘗試不同的方法，同時保持原始會話完整，請改用 fork（分支）。

常見使用情境 Common Use Cases

Checkpoints 在以下情況下特別有用：

  • 探索替代方案：嘗試不同的實作方法，而不會遺失起點
  • 從錯誤中恢復：快速撤銷引入錯誤或破壞功能的變更
  • 迭代功能：進行變體實驗，知道您可以恢復到正常運作的狀態
  • 釋放上下文空間：從中間點開始摘要冗長的除錯會話，保持初始說明完整

限制 Limitations

Bash 命令變更未追蹤 Bash Command Changes Not Tracked

Checkpointing 不追蹤由 bash 命令修改的檔案。例如，如果 Claude Code 執行：

rm file.txt
mv old.txt new.txt
cp source.txt dest.txt

這些檔案修改無法透過回溯撤銷。只有透過 Claude 的檔案編輯工具進行的直接檔案編輯才會被追蹤。

外部變更未追蹤 External Changes Not Tracked

Checkpointing 僅追蹤在目前會話中編輯過的檔案。您在 Claude Code 外部對檔案所做的手動變更以及來自其他並行會話的編輯通常不會被擷取。

不是版本控制的替代品 Not a Replacement for Version Control

Checkpoints 設計用於快速的會話級恢復。對於永久版本歷史和協作：

  • 繼續使用版本控制（例如 Git）進行提交、分支和長期歷史
  • Checkpoints 補充但不替代適當的版本控制
  • 將 checkpoints 視為「本地撤銷」，將 Git 視為「永久歷史」

Bash 工具行為 Bash Tool Behavior

Bash 工具在單獨的處理程序中執行每個命令，具有以下持久性行為：

  • 工作目錄在命令之間保持不變。設定 CLAUDE_BASH_MAINTAIN_PROJECT_WORKING_DIR=1 以在每個命令後重置為專案目錄
  • 環境變數不持久。一個命令中的 export 在下一個命令中將不可用

在啟動 Claude Code 之前啟動您的 virtualenv 或 conda 環境。要使環境變數在 Bash 命令之間保持不變，請在啟動 Claude Code 之前將 CLAUDE_ENV_FILE 設定為 shell 腳本，或使用 SessionStart hook 動態填充它。

輸出樣式 Output Styles

將 Claude Code 適配用於軟體工程之外的用途。輸出樣式讓您可以將 Claude Code 用作任何類型的代理，同時保留其核心功能，例如執行本地腳本、讀取/寫入檔案和追蹤 TODO。

內建輸出樣式 Built-in Output Styles

Claude Code 的 default（預設） 輸出樣式是現有的系統提示，旨在幫助您高效地完成軟體工程任務。還有兩種額外的內建輸出樣式，專注於教您了解程式碼庫和 Claude 的運作方式：

  • Explanatory（解釋型）：在幫助您完成軟體工程任務的同時提供教育性的「Insights（洞察）」。幫助您理解實作選擇和程式碼庫模式。
  • Learning（學習型）：協作式的邊學邊做模式，Claude 不僅會在編碼時分享「Insights」，還會要求您自己貢獻小的、策略性的程式碼片段。Claude Code 將在您的程式碼中新增 TODO(human) 標記供您實作。

輸出樣式如何運作 How Output Styles Work

輸出樣式直接修改 Claude Code 的系統提示。所有輸出樣式都排除了高效輸出的說明。自訂輸出樣式排除了編碼說明，除非 keep-coding-instructions 為 true。所有輸出樣式都在系統提示的末尾新增了自己的自訂說明。

變更您的輸出樣式 Change Your Output Style

執行 /config 並選擇 Output Style（輸出樣式） 從選單中選擇一種樣式。您的選擇會儲存到本地專案層級的 .claude/settings.local.json。

要在不使用選單的情況下設定樣式，直接編輯設定檔中的 outputStyle 欄位：

{
  "outputStyle": "Explanatory"
}

由於輸出樣式是在會話開始時在系統提示中設定的，變更將在您下次啟動新會話時生效。

自訂輸出樣式是包含 frontmatter 和將新增到系統提示的文字的 Markdown 檔案。您可以在使用者層級（~/.claude/output-styles）或專案層級（.claude/output-styles）儲存這些檔案。

設定您的模型 Set Your Model

您可以透過多種方式設定模型，按優先順序列出：

  1. 在會話期間 - 使用 /model <alias|name> 在會話中途切換模型
  2. 啟動時 - 使用 claude --model <alias|name> 啟動
  3. 環境變數 - 設定 ANTHROPIC_MODEL=<alias|name>
  4. 設定 - 在設定檔中使用 model 欄位永久設定

使用範例：

# 使用 Opus 啟動
claude --model opus

# 在會話期間切換到 Sonnet
/model sonnet

調整工作量級別 Adjust Effort Level

工作量級別控制自適應推理，根據任務複雜性動態分配思考。較低的工作量對於直接任務更快更便宜，而較高的工作量為複雜問題提供更深入的推理。

三個級別在會話中持續存在：low、medium 和 high。第四個級別 max 提供最深入的推理，對 token 支出沒有限制，因此回應速度更慢，成本比 high 更高。max 僅在 Opus 4.6 上可用。

Opus 4.6 和 Sonnet 4.6 預設為中等工作量。中等是大多數編碼任務的推薦級別：它平衡了速度和推理深度，更高的級別可能導致模型過度思考日常工作。為真正受益於更深入推理的任務保留 high 或 max。

設定工作量：

  • /effort：執行 /effort low、/effort medium、/effort high 或 /effort max 來變更級別
  • 在 /model 中：選擇模型時使用左右方向鍵調整工作量滑桿
  • --effort 旗標：在啟動 Claude Code 時傳入相應的值
  • 環境變數：設定 CLAUDE_CODE_EFFORT_LEVEL
  • 設定：在設定檔中設定 effortLevel

Slack 中的 Claude Code - Claude Code in Slack

直接從 Slack 工作區委派編碼任務。Slack 中的 Claude Code 將 Claude Code 的強大功能直接引入您的 Slack 工作區。當您使用編碼任務提及 @Claude 時，Claude 會自動偵測意圖並在網頁上建立 Claude Code 會話。

使用情境 Use Cases

  • Bug 調查和修復：要求 Claude 在 Slack 頻道中報告 Bug 時立即調查和修復
  • 快速程式碼審查和修改：讓 Claude 根據團隊回饋實作小功能或重構程式碼
  • 協作除錯：當團隊討論提供關鍵背景資訊時，Claude 可以使用該資訊來指導其除錯方法
  • 並行任務執行：在 Slack 中啟動編碼任務，同時繼續其他工作

前置條件 Prerequisites

要求 詳情
Claude 方案 Pro、Max、Team 或 Enterprise，具有 Claude Code 存取權限
網頁上的 Claude Code 必須啟用對網頁上的 Claude Code 的存取
GitHub 帳號 連接到網頁上的 Claude Code，至少有一個儲存庫已驗證
Slack 驗證 您的 Slack 帳號透過 Claude 應用程式連結到您的 Claude 帳號

在 Slack 中設定 Claude Code Set Up Claude Code in Slack

步驟 1：在 Slack 中安裝 Claude 應用程式

工作區管理員必須從 Slack 應用程式市場安裝 Claude 應用程式。存取 Slack 應用程式市場並點擊「Add to Slack」開始安裝程序。

步驟 2：連接您的 Claude 帳號

安裝應用程式後，驗證您的個人 Claude 帳號：

  1. 透過點擊您的應用程式部分中的「Claude」在 Slack 中開啟 Claude 應用程式
  2. 導覽到應用程式主頁頁籤
  3. 點擊「Connect」將您的 Slack 帳號與您的 Claude 帳號連結
  4. 在瀏覽器中完成驗證流程

步驟 3：設定網頁上的 Claude Code

確保您網頁上的 Claude Code 已正確設定：

  • 存取 claude.ai/code 並使用您連接到 Slack 的同一帳號登入
  • 如果尚未連接，請連接您的 GitHub 帳號
  • 驗證至少一個您希望 Claude 使用的儲存庫

步驟 4：選擇您的路由模式

連接您的帳號後，設定 Claude 如何在 Slack 中處理您的訊息。導覽到 Slack 中的 Claude 應用程式主頁以找到 路由模式 設定。

模式 行為
Code Only（僅程式碼） Claude 將所有 @mentions 路由到 Claude Code 會話。最適合僅將 Claude 用於 Slack 中開發任務的團隊
Code + Chat（程式碼 + 聊天） Claude 分析每條訊息並在 Claude Code（用於編碼任務）和 Claude Chat（用於寫作、分析和常見問題）之間智慧路由
注意：在 Code + Chat 模式下，如果 Claude 將訊息路由到聊天但您想要編碼會話，您可以點擊「Retry as Code」來建立 Claude Code 會話。

運作方式 How It Works

當您在 Slack 頻道或執行緒中提及 @Claude 時，Claude 會自動分析您的訊息以確定它是否是編碼任務。如果 Claude 偵測到編碼意圖，它將把您的請求路由到網頁上的 Claude Code。

警告：當在 Slack 中呼叫 @Claude 時，Claude 可以存取對話上下文以更好地理解您的請求。Claude 可能會遵循上下文中其他訊息的指示，因此使用者應確保僅在受信任的 Slack 對話中使用 Claude。

會話流程 Session Flow

  1. 啟動：您使用編碼請求 @mention Claude
  2. 偵測：Claude 分析您的訊息並偵測編碼意圖
  3. 會話建立：在 claude.ai/code 上建立新的 Claude Code 會話
  4. 進度更新：Claude 在工作進行時向您的 Slack 執行緒發布狀態更新
  5. 完成：完成後，Claude @mentions 您並提供摘要和操作按鈕
  6. 審查：點擊「View Session」查看完整記錄，或點擊「Create PR」開啟 pull request

最佳實踐 Best Practices

撰寫有效的請求

  • 具體說明：在相關時包括檔名、函式名或錯誤訊息
  • 提供上下文：如果從對話中不清楚，請提及儲存庫或專案
  • 定義成功：解釋「完成」的樣子 -- Claude 應該撰寫測試嗎？更新文件？
  • 使用執行緒：在討論 Bug 或功能時在執行緒中回覆

程式碼審查 Code Review

設定自動化 PR 審查，透過對完整程式碼庫的多代理分析來捕獲邏輯錯誤、安全漏洞和迴歸問題。

注意：Code Review 處於研究預覽階段，僅適用於 Teams 和 Enterprise 訂閱。

Code Review 分析您的 GitHub pull request，並在發現問題的程式碼行上發布內嵌評論。一支由專業代理組成的團隊在完整程式碼庫的上下文中檢查程式碼變更，尋找邏輯錯誤、安全漏洞、破損的邊界情況和微妙的迴歸問題。

審查運作方式 How Reviews Work

一旦管理員為您的組織啟用 Code Review，審查將在 PR 開啟時、每次推送時或手動請求時觸發，具體取決於儲存庫的設定行為。在任何模式下，註解 @claude review 可以在 PR 上啟動審查。

當審查執行時，多個代理在 Anthropic 基礎設施上並行分析差異和周圍程式碼。每個代理尋找不同類別的問題，然後驗證步驟檢查候選項是否與實際程式碼行為相符，以過濾掉誤報。結果被去重、按嚴重程度排序，並作為內嵌評論發布。

嚴重程度級別 Severity Levels

標記 嚴重程度 含義
🔴 Normal（正常） 應在合併前修復的錯誤
🟡 Minor（小問題） 輕微問題，值得修復但不阻止
🟣 Pre-existing（預先存在） 程式碼庫中存在但不是由此 PR 引入的錯誤

設定 Code Review Set Up Code Review

步驟 1：開啟 Claude Code 管理員設定

前往 claude.ai/admin-settings/claude-code 並找到 Code Review 部分。您需要對 Claude 組織具有管理員存取權限。

步驟 2：開始設定

點擊 Setup（設定）。這將開始 GitHub App 安裝流程。

步驟 3：安裝 Claude GitHub App

按照提示將 Claude GitHub App 安裝到您的 GitHub 組織。該應用程式請求這些儲存庫權限：

  • Contents（內容）：讀寫
  • Issues（議題）：讀寫
  • Pull requests：讀寫

步驟 4：選擇儲存庫

選擇要為 Code Review 啟用的儲存庫。您可以稍後新增更多儲存庫。

步驟 5：為每個儲存庫設定審查觸發器

設定完成後，Code Review 部分在表格中顯示您的儲存庫。對於每個儲存庫，使用 Review Behavior（審查行為） 下拉選單選擇何時執行審查：

  • Once after PR creation（PR 建立後一次）：當 PR 開啟或標記為準備審查時執行一次審查
  • After every push（每次推送後）：在每次推送到 PR 分支時執行審查
  • Manual（手動）：僅當有人在 PR 上註解 @claude review 時才啟動審查

手動觸發審查 Manually Trigger Reviews

在 pull request 上註解 @claude review 以啟動審查。要使註解觸發審查：

  • 將其作為頂層 PR 評論發布，而不是差異行上的內嵌評論
  • 在註解開頭放置 @claude review
  • 您必須對儲存庫具有所有者、成員或協作者存取權限
  • PR 必須開啟且不是草稿

自訂審查 Customize Reviews

Code Review 從您的儲存庫讀取兩個檔案來指導它標記的內容：

  • CLAUDE.md：共享專案說明，Claude Code 用於所有任務。Code Review 讀取您的儲存庫的 CLAUDE.md 檔案，並將新引入的違規視為小問題級別的發現。
  • REVIEW.md：僅審查指導，在程式碼審查期間專門讀取。使用它來編碼公司或團隊風格指南、語言或框架特定的慣例，以及 Claude 應始終標記或跳過的內容。

範例 REVIEW.md：

# Code Review Guidelines

## Always check
- New API endpoints have corresponding integration tests
- Database migrations are backward-compatible
- Error messages don't leak internal details to users

## Style
- Prefer `match` statements over chained `isinstance` checks
- Use structured logging, not f-string interpolation in log calls

## Skip
- Generated files under `src/gen/`
- Formatting-only changes in `*.lock` files

檢視使用情況 View Usage

前往 claude.ai/analytics/code-review 以檢視整個組織的 Code Review 活動。儀表板顯示審查的 PR 計數、每週成本、因開發人員解決問題而自動解決的審查評論計數，以及按儲存庫的分解。

定價 Pricing

Code Review 根據 token 使用情況計費。審查平均花費 $15-25，隨 PR 大小、程式碼庫複雜性和需要驗證的問題數量而擴展。Code Review 使用透過額外用量單獨計費，不計入您的方案包含的使用。

您選擇的審查觸發器影響總成本：

  • Once after PR creation（PR 建立後一次）：每個 PR 執行一次
  • After every push（每次推送後）：在每次推送時執行，將成本乘以推送次數
  • Manual（手動）：在有人在 PR 上註解 @claude review 之前沒有審查
