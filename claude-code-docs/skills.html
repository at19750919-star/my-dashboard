Skills・MCP・Hooks

自訂技能建立、MCP 伺服器整合、事件驅動鉤子、動態上下文注入——把 AI 變成「你自己的 AI」的開關都在這。

使用 Skills 擴展 Claude

建立、管理和分享 skills，在 Claude Code 中擴展 Claude 的功能。包括自訂指令和捆綁 skills。

Skills 擴展了 Claude 能做的事情。建立一個 SKILL.md 檔案，其中包含指示，Claude 會將其加入工具包中。Claude 在相關時使用 skills，或者你可以使用 /skill-name 直接呼叫。

注意： 對於內建指令（如 /help 和 /compact），請參閱內建指令參考。

自訂指令已合併到 skills 中。 .claude/commands/deploy.md 中的檔案和 .claude/skills/deploy/SKILL.md 中的 skill 都會建立 /deploy 並以相同的方式運作。你現有的 .claude/commands/ 檔案繼續運作。Skills 增加了可選功能：支援檔案的目錄、控制你或 Claude 是否呼叫它們的 frontmatter，以及 Claude 在相關時自動載入它們的能力。

Claude Code skills 遵循 Agent Skills 開放標準，該標準適用於多個 AI 工具。Claude Code 使用額外功能擴展了該標準，如呼叫控制、subagent 執行和動態上下文注入。

捆綁 Skills

捆綁 skills 隨 Claude Code 一起提供，在每個工作階段中都可用。與內建指令不同，內建指令直接執行固定邏輯，捆綁 skills 是基於提示的：它們為 Claude 提供詳細的劇本，讓它使用其工具來編排工作。這意味著捆綁 skills 可以生成並行代理、讀取檔案並適應你的程式碼庫。

你呼叫捆綁 skills 的方式與呼叫任何其他 skill 相同：輸入 / 後跟 skill 名稱。在下表中，<arg> 表示必需的參數，[arg] 表示可選參數。

Skill目的
/batch <instruction>在程式碼庫中並行編排大規模更改。研究程式碼庫，將工作分解為 5 到 30 個獨立單元，並呈現計畫。獲得批准後，在隔離的 git worktree 中為每個單元生成一個背景代理。每個代理實作其單元、執行測試並開啟 pull request。需要 git 存儲庫。範例：/batch migrate src/ from Solid to React
/claude-api為你的專案語言（Python、TypeScript、Java、Go、Ruby、C#、PHP 或 cURL）載入 Claude API 參考資料，以及 Python 和 TypeScript 的 Agent SDK 參考。涵蓋工具使用、串流、批次處理、結構化輸出和常見陷阱。當你的程式碼匯入 anthropic、@anthropic-ai/sdk 或 claude_agent_sdk 時也會自動啟用
/debug [description]為目前工作階段啟用除錯日誌記錄並透過讀取工作階段除錯日誌來排查問題。預設情況下除錯日誌處於關閉狀態，除非你使用 claude --debug 啟動，因此在工作階段中期執行 /debug 會從該點開始擷取日誌。可選地描述問題以集中分析
/loop [interval] <prompt>在工作階段保持開啟狀態時按間隔重複執行提示。適用於輪詢部署、監督 PR 或定期重新執行另一個 skill。範例：/loop 5m check if the deploy finished
/simplify [focus]查看你最近更改的檔案以尋找程式碼重用、品質和效率問題，然後修復它們。並行生成三個審查代理，彙總其發現並套用修復。傳遞文字以專注於特定問題：/simplify focus on memory efficiency

入門

建立你的第一個 Skill

此範例建立一個 skill，教 Claude 使用視覺圖表和類比來解釋程式碼。由於它使用預設 frontmatter，Claude 可以在你詢問某事如何運作時自動載入它，或者你可以使用 /explain-code 直接呼叫它。

步驟 1：建立 skill 目錄

在你的個人 skills 資料夾中為 skill 建立一個目錄。個人 skills 在你的所有專案中都可用。

mkdir -p ~/.claude/skills/explain-code

步驟 2：編寫 SKILL.md

每個 skill 都需要一個 SKILL.md 檔案，包含兩部分：YAML frontmatter（在 --- 標記之間）告訴 Claude 何時使用該 skill，以及包含 Claude 在呼叫該 skill 時遵循的指示的 markdown 內容。name 欄位變成 /slash-command，description 幫助 Claude 決定何時自動載入它。

建立 ~/.claude/skills/explain-code/SKILL.md：

---
name: explain-code
description: Explains code with visual diagrams and analogies. Use when explaining how code works, teaching about a codebase, or when the user asks "how does this work?"
---

When explaining code, always include:

1. **Start with an analogy**: Compare the code to something from everyday life
2. **Draw a diagram**: Use ASCII art to show the flow, structure, or relationships
3. **Walk through the code**: Explain step-by-step what happens
4. **Highlight a gotcha**: What's a common mistake or misconception?

Keep explanations conversational. For complex concepts, use multiple analogies.

步驟 3：測試 skill

你可以透過兩種方式測試它：

讓 Claude 自動呼叫它，透過詢問與描述匹配的內容：

How does this code work?

或直接使用 skill 名稱呼叫它：

/explain-code src/auth/login.ts

無論哪種方式，Claude 都應該在其解釋中包含類比和 ASCII 圖表。

Skills 的位置

你儲存 skill 的位置決定了誰可以使用它：

位置路徑適用於
企業請參閱託管設定你的組織中的所有使用者
個人~/.claude/skills/<skill-name>/SKILL.md你的所有專案
專案.claude/skills/<skill-name>/SKILL.md僅此專案
外掛<plugin>/skills/<skill-name>/SKILL.md啟用外掛的位置

當 skills 在各個層級共享相同的名稱時，更高優先權的位置獲勝：企業 > 個人 > 專案。外掛 skills 使用 plugin-name:skill-name 命名空間，因此它們不能與其他層級衝突。如果你在 .claude/commands/ 中有檔案，它們的運作方式相同，但如果 skill 和指令共享相同的名稱，skill 優先。

從巢狀目錄自動發現

當你在子目錄中處理檔案時，Claude Code 會自動從巢狀的 .claude/skills/ 目錄中發現 skills。例如，如果你正在編輯 packages/frontend/ 中的檔案，Claude Code 也會在 packages/frontend/.claude/skills/ 中尋找 skills。這支援 monorepo 設定，其中套件有自己的 skills。

每個 skill 都是一個以 SKILL.md 作為入口點的目錄：

my-skill/
├── SKILL.md           # 主要指示（必需）
├── template.md        # Claude 要填寫的範本
├── examples/
│   └── sample.md      # 顯示預期格式的範例輸出
└── scripts/
    └── validate.sh    # Claude 可以執行的腳本

SKILL.md 包含主要指示，是必需的。其他檔案是可選的，讓你建構更強大的 skills：Claude 要填寫的範本、顯示預期格式的範例輸出、Claude 可以執行的腳本或詳細的參考文件。從你的 SKILL.md 中引用支援檔案，以便 Claude 知道每個檔案包含什麼以及何時載入它。

.claude/commands/ 中的檔案仍然有效，並支援相同的 frontmatter。建議使用 Skills，因為它們支援額外的功能，如支援檔案。
來自其他目錄的 Skills

在透過 --add-dir 新增的目錄中的 .claude/skills/ 中定義的 skills 會自動載入並由即時更改偵測拾取，因此你可以在工作階段期間編輯它們而無需重新啟動。

來自 --add-dir 目錄的 CLAUDE.md 檔案預設不載入。要載入它們，請設定 CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1。

設定 Skills

Skills 透過 SKILL.md 頂部的 YAML frontmatter 和隨後的 markdown 內容進行設定。

Skill 內容的類型

Skill 檔案可以包含任何指示，但思考你想如何呼叫它們有助於指導要包含的內容：

參考內容 新增 Claude 應用於你目前工作的知識。慣例、模式、風格指南、領域知識。此內容內聯執行，以便 Claude 可以將其與你的對話上下文一起使用。

---
name: api-conventions
description: API design patterns for this codebase
---

When writing API endpoints:
- Use RESTful naming conventions
- Return consistent error formats
- Include request validation

任務內容 為 Claude 提供特定操作的逐步指示，如部署、提交或程式碼產生。這些通常是你想使用 /skill-name 直接呼叫的操作，而不是讓 Claude 決定何時執行它們。加入 disable-model-invocation: true 以防止 Claude 自動觸發它。

---
name: deploy
description: Deploy the application to production
context: fork
disable-model-invocation: true
---

Deploy the application:
1. Run the test suite
2. Build the application
3. Push to the deployment target

Frontmatter 參考

除了 markdown 內容外，你可以使用 SKILL.md 檔案頂部 --- 標記之間的 YAML frontmatter 欄位來設定 skill 行為：

---
name: my-skill
description: What this skill does
disable-model-invocation: true
allowed-tools: Read, Grep
---

Your skill instructions here...

所有欄位都是可選的。建議使用 description，以便 Claude 知道何時使用該 skill。

欄位必需描述
name否Skill 的顯示名稱。如果省略，使用目錄名稱。僅小寫字母、數字和連字號（最多 64 個字元）。
description建議Skill 的功能以及何時使用它。Claude 使用它來決定何時套用該 skill。如果省略，使用 markdown 內容的第一段。
argument-hint否自動完成期間顯示的提示，指示預期的參數。範例：[issue-number] 或 [filename] [format]。
disable-model-invocation否設為 true 以防止 Claude 自動載入此 skill。用於你想使用 /name 手動觸發的工作流程。預設值：false。
user-invocable否設為 false 以從 / 選單中隱藏。用於使用者不應直接呼叫的背景知識。預設值：true。
allowed-tools否當此 skill 處於活動狀態時，Claude 可以使用而無需請求權限的工具。
model否當此 skill 處於活動狀態時要使用的模型。
effort否當此 skill 處於活動狀態時的工作量級別。覆蓋工作階段工作量級別。預設值：繼承自工作階段。選項：low、medium、high、max（僅 Opus 4.6）。
context否設為 fork 以在分叉的 subagent 上下文中執行。
agent否當設定 context: fork 時要使用的 subagent 類型。
hooks否限定於此 skill 生命週期的 hooks。
可用的字串替換

Skills 支援 skill 內容中動態值的字串替換：

變數描述
$ARGUMENTS呼叫 skill 時傳遞的所有參數。如果內容中不存在 $ARGUMENTS，參數將作為 ARGUMENTS: <value> 附加。
$ARGUMENTS[N]按 0 基索引存取特定參數，如 $ARGUMENTS[0] 表示第一個參數。
$N$ARGUMENTS[N] 的簡寫，如 $0 表示第一個參數或 $1 表示第二個參數。
${CLAUDE_SESSION_ID}目前工作階段 ID。適用於日誌記錄、建立工作階段特定檔案或將 skill 輸出與工作階段關聯。
${CLAUDE_SKILL_DIR}包含 skill 的 SKILL.md 檔案的目錄。在 bash 注入指令中使用它來引用與 skill 捆綁的腳本或檔案，無論目前工作目錄如何。

使用替換的範例：

---
name: session-logger
description: Log activity for this session
---

Log the following to logs/${CLAUDE_SESSION_ID}.log:

$ARGUMENTS

新增支援檔案

Skills 可以在其目錄中包含多個檔案。這使 SKILL.md 專注於要點，同時讓 Claude 僅在需要時存取詳細的參考資料。大型參考文件、API 規格或範例集合不需要在每次 skill 執行時載入到上下文中。

my-skill/
├── SKILL.md (required - overview and navigation)
├── reference.md (detailed API docs - loaded when needed)
├── examples.md (usage examples - loaded when needed)
└── scripts/
    └── helper.py (utility script - executed, not loaded)

從 SKILL.md 中引用支援檔案，以便 Claude 知道每個檔案包含什麼以及何時載入它：

## Additional resources

- For complete API details, see [reference.md](reference.md)
- For usage examples, see [examples.md](examples.md)
將 SKILL.md 保持在 500 行以下。將詳細的參考資料移到單獨的檔案中。

控制誰呼叫 Skill

預設情況下，你和 Claude 都可以呼叫任何 skill。你可以輸入 /skill-name 直接呼叫它，Claude 可以在與你的對話相關時自動載入它。兩個 frontmatter 欄位讓你限制這一點：

  • disable-model-invocation: true：只有你可以呼叫該 skill。用於有副作用的工作流程或你想控制時間的工作流程，如 /commit、/deploy 或 /send-slack-message。你不希望 Claude 因為你的程式碼看起來準備好了就決定部署。
  • user-invocable: false：只有 Claude 可以呼叫該 skill。用於不可作為指令操作的背景知識。legacy-system-context skill 解釋了舊系統的工作原理。Claude 在相關時應該知道這一點，但 /legacy-system-context 對使用者來說不是一個有意義的操作。

以下是兩個欄位如何影響呼叫和上下文載入：

Frontmatter你可以呼叫Claude 可以呼叫何時載入到上下文中
（預設）是是描述始終在上下文中，呼叫時載入完整 skill
disable-model-invocation: true是否描述不在上下文中，你呼叫時載入完整 skill
user-invocable: false否是描述始終在上下文中，呼叫時載入完整 skill
在一般工作階段中，skill 描述被載入到上下文中，以便 Claude 知道什麼可用，但完整 skill 內容僅在呼叫時載入。預載入 skills 的 subagents 的運作方式不同：完整 skill 內容在啟動時注入。

限制工具存取

使用 allowed-tools 欄位來限制當 skill 處於活動狀態時 Claude 可以使用哪些工具。此 skill 建立一個唯讀模式，其中 Claude 可以瀏覽檔案但不能修改它們：

---
name: safe-reader
description: Read files without making changes
allowed-tools: Read, Grep, Glob
---

將參數傳遞給 Skills

你和 Claude 都可以在呼叫 skill 時傳遞參數。參數可透過 $ARGUMENTS 佔位符取得。

此 skill 按編號修復 GitHub issue。$ARGUMENTS 佔位符被替換為 skill 名稱後面的任何內容：

---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
---

Fix GitHub issue $ARGUMENTS following our coding standards.

1. Read the issue description
2. Understand the requirements
3. Implement the fix
4. Write tests
5. Create a commit

當你執行 /fix-issue 123 時，Claude 收到 "Fix GitHub issue 123 following our coding standards..."

如果你使用參數呼叫 skill 但 skill 不包含 $ARGUMENTS，Claude Code 會將 ARGUMENTS: <your input> 附加到 skill 內容的末尾，以便 Claude 仍然看到你輸入的內容。

要按位置存取單個參數，使用 $ARGUMENTS[N] 或較短的 $N：

---
name: migrate-component
description: Migrate a component from one framework to another
---

Migrate the $ARGUMENTS[0] component from $ARGUMENTS[1] to $ARGUMENTS[2].
Preserve all existing behavior and tests.

執行 /migrate-component SearchBar React Vue 會將 $ARGUMENTS[0] 替換為 SearchBar，$ARGUMENTS[1] 替換為 React，$ARGUMENTS[2] 替換為 Vue。

進階模式

注入動態上下文

`` !`<command>` `` 語法在將 skill 內容傳送給 Claude 之前執行 shell 指令。指令輸出替換佔位符，因此 Claude 接收實際資料，而不是指令本身。

此 skill 透過使用 GitHub CLI 取得即時 PR 資料來摘要 pull request。`` !`gh pr diff` `` 和其他指令首先執行，其輸出被插入到提示中：

---
name: pr-summary
description: Summarize changes in a pull request
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Pull request context
- PR diff: !`gh pr diff`
- PR comments: !`gh pr view --comments`
- Changed files: !`gh pr diff --name-only`

## Your task
Summarize this pull request...

當此 skill 執行時：

  1. 每個 `` !`<command>` `` 立即執行（在 Claude 看到任何內容之前）
  2. 輸出替換 skill 內容中的佔位符
  3. Claude 接收帶有實際 PR 資料的完全呈現的提示

這是預處理，不是 Claude 執行的內容。Claude 只看到最終結果。

要在 skill 中啟用擴展思考 (extended thinking)，在你的 skill 內容中的任何地方包含單詞 "ultrathink"。

在 Subagent 中執行 Skills

當你想讓 skill 在隔離中執行時，在你的 frontmatter 中加入 context: fork。skill 內容變成驅動 subagent 的提示。它將無法存取你的對話歷史。

警告： context: fork 僅對具有明確指示的 skills 有意義。如果你的 skill 包含「使用這些 API 慣例」之類的指引而沒有任務，subagent 會收到指引但沒有可操作的提示，並返回而沒有有意義的輸出。

Skills 和 subagents 以兩個方向協同工作：

方法系統提示任務也載入
帶有 context: fork 的 Skill來自代理類型（Explore、Plan 等）SKILL.md 內容CLAUDE.md
帶有 skills 欄位的 SubagentSubagent 的 markdown 正文Claude 的委派訊息預載入的 skills + CLAUDE.md

限制 Claude 的 Skill 存取

預設情況下，Claude 可以呼叫任何沒有設定 disable-model-invocation: true 的 skill。

控制 Claude 可以呼叫哪些 skills 的三種方式：

透過在 /permissions 中拒絕 Skill 工具來停用所有 skills：

# Add to deny rules:
Skill

使用權限規則允許或拒絕特定 skills：

# Allow only specific skills
Skill(commit)
Skill(review-pr *)

# Deny specific skills
Skill(deploy *)

權限語法：Skill(name) 用於精確匹配，Skill(name *) 用於帶有任何參數的前綴匹配。

透過在其 frontmatter 中加入 disable-model-invocation: true 來隱藏單個 skills。這會從 Claude 的上下文中完全移除該 skill。

分享 Skills

Skills 可以根據你的受眾在不同範圍內分發：

  • 專案 skills：將 .claude/skills/ 提交到版本控制
  • 外掛：在你的外掛中建立 skills/ 目錄
  • 託管：透過託管設定部署組織範圍內

產生視覺輸出

Skills 可以捆綁並執行任何語言的腳本，為 Claude 提供單個提示中不可能的功能。一個強大的模式是產生視覺輸出：在瀏覽器中開啟的互動式 HTML 檔案，用於探索資料、除錯或建立報告。

疑難排解

Skill 未觸發

如果 Claude 在預期時不使用你的 skill：

  1. 檢查描述是否包含使用者會自然說的關鍵字
  2. 驗證 skill 是否出現在 What skills are available? 中
  3. 嘗試重新表述你的請求以更接近描述
  4. 如果 skill 是使用者可呼叫的，使用 /skill-name 直接呼叫它

Skill 觸發過於頻繁

如果 Claude 在你不想要時使用你的 skill：

  1. 使描述更具體
  2. 如果你只想手動呼叫，加入 disable-model-invocation: true

Claude 看不到我的所有 Skills

Skill 描述被載入到上下文中，以便 Claude 知道什麼可用。如果你有許多 skills，它們可能會超過字元預算。預算在上下文視窗的 2% 處動態擴展，回退為 16,000 個字元。執行 /context 以檢查有關排除的 skills 的警告。

要覆蓋限制，設定 SLASH_COMMAND_TOOL_CHAR_BUDGET 環境變數。

透過 MCP 將 Claude Code 連接到工具

了解如何使用 Model Context Protocol (MCP) 將 Claude Code 連接到你的工具。

Claude Code 可以透過 Model Context Protocol (MCP)（一個用於 AI 工具整合的開源標準）連接到數百個外部工具和資料來源。MCP 伺服器為 Claude Code 提供對你的工具、資料庫和 API 的存取權限。

使用 MCP 可以做什麼

連接 MCP 伺服器後，你可以要求 Claude Code：

  • 從問題追蹤器實作功能：「加入 JIRA issue ENG-4521 中描述的功能，並在 GitHub 上建立 PR。」
  • 分析監控資料：「檢查 Sentry 和 Statsig 以檢查 ENG-4521 中描述的功能的使用情況。」
  • 查詢資料庫：「根據我們的 PostgreSQL 資料庫，找出使用功能 ENG-4521 的 10 個隨機使用者的電子郵件。」
  • 整合設計：「根據在 Slack 中發布的新 Figma 設計更新我們的標準電子郵件範本」
  • 自動化工作流程：「建立 Gmail 草稿，邀請這 10 個使用者參加關於新功能的回饋會議。」
  • 對外部事件做出反應：MCP 伺服器也可以充當 channels（頻道），將訊息推送到你的工作階段中，因此當你不在時，Claude 可以對 Telegram 訊息、Discord 聊天或 webhook 事件做出反應。

安裝 MCP 伺服器

MCP 伺服器可以根據你的需求以三種不同的方式進行設定：

選項 1：新增遠端 HTTP 伺服器

HTTP 伺服器是連接到遠端 MCP 伺服器的建議選項。這是雲端服務最廣泛支援的傳輸方式。

# 基本語法
claude mcp add --transport http <name> <url>

# 真實範例：連接到 Notion
claude mcp add --transport http notion https://mcp.notion.com/mcp

# 帶有 Bearer token 的範例
claude mcp add --transport http secure-api https://api.example.com/mcp \
  --header "Authorization: Bearer your-token"

選項 2：新增遠端 SSE 伺服器

警告： SSE (Server-Sent Events) 傳輸已棄用。請在可用的地方使用 HTTP 伺服器。 
# 基本語法
claude mcp add --transport sse <name> <url>

# 真實範例：連接到 Asana
claude mcp add --transport sse asana https://mcp.asana.com/sse

選項 3：新增本機 stdio 伺服器

Stdio 伺服器作為你機器上的本機程序執行。它們非常適合需要直接系統存取或自訂腳本的工具。

# 基本語法
claude mcp add [options] <name> -- <command> [args...]

# 真實範例：新增 Airtable 伺服器
claude mcp add --transport stdio --env AIRTABLE_API_KEY=YOUR_KEY airtable \
  -- npx -y airtable-mcp-server
重要：選項順序

所有選項（--transport、--env、--scope、--header）必須在伺服器名稱之前。然後 --（雙破折號）將伺服器名稱與傳遞給 MCP 伺服器的指令和參數分開。

管理你的伺服器

設定後，你可以使用這些指令管理你的 MCP 伺服器：

# 列出所有已設定的伺服器
claude mcp list

# 取得特定伺服器的詳細資訊
claude mcp get github

# 移除伺服器
claude mcp remove github

# （在 Claude Code 中）檢查伺服器狀態
/mcp

動態工具更新

Claude Code 支援 MCP list_changed 通知，允許 MCP 伺服器動態更新其可用工具、提示和資源，而無需你中斷連線並重新連線。當 MCP 伺服器傳送 list_changed 通知時，Claude Code 會自動重新整理來自該伺服器的可用功能。

使用 Channels 推送訊息

MCP 伺服器也可以直接將訊息推送到你的工作階段中，以便 Claude 可以對外部事件（如 CI 結果、監控警報或聊天訊息）做出反應。要啟用此功能，你的伺服器宣告 claude/channel 功能，並在啟動時使用 --channels 旗標選擇加入。

提示：

  • 使用 --scope 旗標指定設定的儲存位置：
    • local（預設）：僅在目前專案中對你可用
    • project：透過 .mcp.json 檔案與專案中的每個人共享
    • user：在所有專案中對你可用
  • 使用 --env 旗標設定環境變數（例如，--env KEY=value）
  • 使用 MCP_TIMEOUT 環境變數設定 MCP 伺服器啟動逾時（例如，MCP_TIMEOUT=10000 claude 設定 10 秒逾時）
  • 當 MCP 工具輸出超過 10,000 個 token 時，Claude Code 將顯示警告。要增加此限制，請設定 MAX_MCP_OUTPUT_TOKENS 環境變數
  • 使用 /mcp 對需要 OAuth 2.0 驗證的遠端伺服器進行驗證

Windows 使用者：在原生 Windows（不是 WSL）上，使用 npx 的本機 MCP 伺服器需要 cmd /c 包裝器以確保正確執行。

claude mcp add --transport stdio my-server -- cmd /c npx -y @some/package

外掛提供的 MCP 伺服器

外掛可以捆綁 MCP 伺服器，在啟用外掛時自動提供工具和整合。外掛 MCP 伺服器的運作方式與使用者設定的伺服器相同。

MCP 安裝範圍

MCP 伺服器可以在三個不同的範圍層級進行設定：

本機範圍

本機範圍的伺服器代表預設設定層級。這些伺服器對你保持私密，僅在目前專案目錄中工作時可存取。此範圍非常適合個人開發伺服器、實驗設定或包含不應共享的敏感憑證的伺服器。

# 新增本機範圍的伺服器（預設）
claude mcp add --transport http stripe https://mcp.stripe.com

# 明確指定本機範圍
claude mcp add --transport http stripe --scope local https://mcp.stripe.com

專案範圍

專案範圍的伺服器透過在專案根目錄中儲存設定在 .mcp.json 檔案中來啟用團隊協作。此檔案設計為簽入版本控制，確保所有團隊成員都可以存取相同的 MCP 工具和服務。

# 新增專案範圍的伺服器
claude mcp add --transport http paypal --scope project https://mcp.paypal.com/mcp

產生的 .mcp.json 檔案遵循標準化格式：

{
  "mcpServers": {
    "shared-server": {
      "command": "/path/to/server",
      "args": [],
      "env": {}
    }
  }
}

出於安全原因，Claude Code 在使用來自 .mcp.json 檔案的專案範圍的伺服器之前會提示批准。如果你需要重置這些批准選擇，請使用 claude mcp reset-project-choices 指令。

使用者範圍

使用者範圍的伺服器儲存在 ~/.claude.json 中，並提供跨專案可存取性，使其在你機器上的所有專案中可用，同時對你的使用者帳戶保持私密。

# 新增使用者伺服器
claude mcp add --transport http hubspot --scope user https://mcp.hubspot.com/anthropic

選擇正確的範圍

根據以下條件選擇你的範圍：

  • 本機範圍：個人伺服器、實驗設定或特定於一個專案的敏感憑證
  • 專案範圍：團隊共享的伺服器、專案特定的工具或協作所需的服務
  • 使用者範圍：跨多個專案需要的個人工具程式、開發工具或經常使用的服務

範圍階層和優先權

MCP 伺服器設定遵循清晰的優先權階層。當具有相同名稱的伺服器存在於多個範圍時，系統透過首先優先考慮本機範圍的伺服器、其次是專案範圍的伺服器，最後是使用者範圍的伺服器來解決衝突。

.mcp.json 中的環境變數擴展

Claude Code 支援 .mcp.json 檔案中的環境變數擴展，允許團隊共享設定，同時為特定於機器的路徑和 API 金鑰等敏感值保持彈性。

支援的語法：

  • ${VAR} - 擴展為環境變數 VAR 的值
  • ${VAR:-default} - 如果設定了 VAR，則擴展為 VAR，否則使用 default
{
  "mcpServers": {
    "api-server": {
      "type": "http",
      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
      "headers": {
        "Authorization": "Bearer ${API_KEY}"
      }
    }
  }
}

使用遠端 MCP 伺服器進行驗證

許多基於雲端的 MCP 伺服器需要驗證。Claude Code 支援 OAuth 2.0 以實現安全連線。

  1. 新增需要驗證的伺服器
  2. 在 Claude Code 中使用 /mcp 指令
  3. 然後按照瀏覽器中的步驟登入

使用固定的 OAuth 回呼連接埠

某些 MCP 伺服器需要預先註冊的特定重導 URI。使用 --callback-port 固定連接埠。

# 使用動態用戶端註冊的固定回呼連接埠
claude mcp add --transport http \
  --callback-port 8080 \
  my-server https://mcp.example.com/mcp

使用預設的 OAuth 憑證

某些 MCP 伺服器不支援自動 OAuth 設定。首先透過伺服器的開發者入口網站註冊 OAuth 應用，然後在新增伺服器時提供憑證。

claude mcp add --transport http \
  --client-id your-client-id --client-secret --callback-port 8080 \
  my-server https://mcp.example.com/mcp

使用動態標頭進行自訂驗證

如果你的 MCP 伺服器使用 OAuth 以外的驗證方案，請使用 headersHelper 在連線時產生請求標頭。

{
  "mcpServers": {
    "internal-api": {
      "type": "http",
      "url": "https://mcp.internal.example.com",
      "headersHelper": "/opt/bin/get-mcp-auth-headers.sh"
    }
  }
}

# 基本語法
claude mcp add-json <name> '<json>'

# 範例：新增帶有 JSON 設定的 HTTP 伺服器
claude mcp add-json weather-api '{"type":"http","url":"https://api.weather.com/mcp","headers":{"Authorization":"Bearer token"}}'

使用來自 Claude.ai 的 MCP 伺服器

如果你已使用 Claude.ai 帳戶登入 Claude Code，你在 Claude.ai 中新增的 MCP 伺服器會自動在 Claude Code 中可用。

要在 Claude Code 中停用 claude.ai MCP 伺服器：

ENABLE_CLAUDEAI_MCP_SERVERS=false claude

將 Claude Code 用作 MCP 伺服器

你可以將 Claude Code 本身用作 MCP 伺服器，其他應用程式可以連接到它：

# 啟動 Claude 作為 stdio MCP 伺服器
claude mcp serve

MCP 輸出限制和警告

  • 輸出警告閾值：當任何 MCP 工具輸出超過 10,000 個 token 時，Claude Code 顯示警告
  • 可設定限制：你可以使用 MAX_MCP_OUTPUT_TOKENS 環境變數調整最大允許的 MCP 輸出 token
  • 預設限制：預設最大值為 25,000 個 token
# 為 MCP 工具輸出設定更高的限制
export MAX_MCP_OUTPUT_TOKENS=50000
claude

回應 MCP 引發請求

MCP 伺服器可以在任務中途使用 elicitation（引發）來請求你的結構化輸入。當伺服器需要無法自行取得的資訊時，Claude Code 會顯示互動式對話框並將你的回應傳遞回伺服器。

伺服器可以透過兩種方式請求輸入：

  • 表單模式：Claude Code 顯示一個對話框，其中包含伺服器定義的表單欄位
  • URL 模式：Claude Code 開啟瀏覽器 URL 以進行驗證或批准

使用 MCP 資源

MCP 伺服器可以公開資源，你可以使用 @ 提及來引用，類似於你引用檔案的方式。

使用格式 @server:protocol://resource/path 來引用資源：

Can you analyze @github:issue://123 and suggest a fix?

使用 MCP 工具搜尋進行擴展

當你設定了許多 MCP 伺服器時，工具定義可能會消耗你的上下文視窗的很大一部分。MCP 工具搜尋透過動態按需載入工具而不是預載入所有工具來解決此問題。

當你的 MCP 工具描述會消耗超過 10% 的上下文視窗時，Claude Code 會自動啟用工具搜尋。

使用 ENABLE_TOOL_SEARCH 環境變數控制工具搜尋行為：

值行為
（未設定）預設啟用
true始終啟用
auto當 MCP 工具超過 10% 的上下文時啟用
auto:<N>在自訂閾值啟用，其中 <N> 是百分比
false停用，所有 MCP 工具預先載入

將 MCP 提示用作指令

MCP 伺服器可以公開在 Claude Code 中作為指令可用的提示。輸入 / 以查看所有可用的指令，包括來自 MCP 伺服器的指令。MCP 提示以 /mcp__servername__promptname 的格式出現。

託管 MCP 設定

對於需要對 MCP 伺服器進行集中控制的組織，Claude Code 支援兩個設定選項：

  1. 使用 managed-mcp.json 的獨占控制：部署使用者無法修改或擴展的固定 MCP 伺服器集
  2. 使用允許清單/拒絕清單的基於策略的控制：允許使用者新增自己的伺服器，但限制允許的伺服器

選項 1：使用 managed-mcp.json 的獨占控制

系統管理員將設定檔部署到系統範圍的目錄：

  • macOS：/Library/Application Support/ClaudeCode/managed-mcp.json
  • Linux 和 WSL：/etc/claude-code/managed-mcp.json
  • Windows：C:\Program Files\ClaudeCode\managed-mcp.json

選項 2：使用允許清單和拒絕清單的基於策略的控制

管理員可以允許使用者設定自己的 MCP 伺服器，同時對允許的伺服器進行限制。此方法在託管設定檔中使用 allowedMcpServers 和 deniedMcpServers。

限制選項：

  1. 按伺服器名稱 (serverName)：匹配伺服器的設定名稱
  2. 按指令 (serverCommand)：匹配用於啟動 stdio 伺服器的確切指令和參數
  3. 按 URL 模式 (serverUrl)：匹配帶有萬用字元支援的遠端伺服器 URL

使用 Hooks 自動化工作流程

當 Claude Code 編輯檔案、完成任務或需要輸入時自動執行 shell 指令。格式化程式碼、傳送通知、驗證指令並強制執行專案規則。

Hooks 是使用者定義的 shell 指令，在 Claude Code 生命週期中的特定點執行。它們對 Claude Code 的行為提供確定性控制，確保某些操作始終發生，而不是依賴 LLM 選擇執行它們。使用 hooks 來強制執行專案規則、自動化重複任務，並將 Claude Code 與現有工具整合。

對於需要判斷而不是確定性規則的決策，你也可以使用基於提示的 hooks (prompt-based hooks) 或基於代理的 hooks (agent-based hooks)，它們使用 Claude 模型來評估條件。

本指南涵蓋常見用例和入門方法。有關完整的事件架構、JSON 輸入/輸出格式和非同步 hooks 及 MCP 工具 hooks 等進階功能，請參閱 Hooks 參考。

設定你的第一個 Hook

要建立 hook，請將 hooks 區塊新增到設定檔。本演練建立一個桌面通知 hook，這樣每當 Claude 等待你的輸入而不是監視終端時，你都會收到警報。

步驟 1：將 hook 新增到你的設定

開啟 ~/.claude/settings.json 並新增一個 Notification hook：

{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude Code needs your attention\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}

步驟 2：驗證設定

輸入 /hooks 開啟 hooks 瀏覽器。你將看到所有可用 hook 事件的清單，每個設定了 hooks 的事件旁邊都有一個計數。

步驟 3：測試 hook

按 Esc 返回 CLI。要求 Claude 做需要權限的事情，然後切換離開終端。你應該會收到桌面通知。

/hooks 選單是唯讀的。要新增、修改或刪除 hooks，請直接編輯你的設定 JSON 或要求 Claude 進行更改。

你可以自動化什麼

Hooks 讓你在 Claude Code 生命週期中的關鍵點執行程式碼：編輯後格式化檔案、在執行前阻止指令、在 Claude 需要輸入時傳送通知、在工作階段開始時注入上下文等。

在 Claude 需要輸入時取得通知

每當 Claude 完成工作並需要你的輸入時取得桌面通知。此 hook 使用 Notification 事件，當 Claude 等待輸入或權限時觸發。

macOS：

{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude Code needs your attention\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}

Windows (PowerShell)：

{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "powershell.exe -Command \"[System.Reflection.Assembly]::LoadWithPartialName('System.Windows.Forms'); [System.Windows.Forms.MessageBox]::Show('Claude Code needs your attention', 'Claude Code')\""
          }
        ]
      }
    ]
  }
}

編輯後自動格式化程式碼

在 Claude 編輯的每個檔案上自動執行 Prettier，以便格式保持一致而無需手動介入。

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

阻止對受保護檔案的編輯

防止 Claude 修改敏感檔案，如 .env、package-lock.json 或 .git/ 中的任何內容。

將腳本儲存到 .claude/hooks/protect-files.sh：

#!/bin/bash
# protect-files.sh

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

PROTECTED_PATTERNS=(".env" "package-lock.json" ".git/")

for pattern in "${PROTECTED_PATTERNS[@]}"; do
  if [[ "$FILE_PATH" == *"$pattern"* ]]; then
    echo "Blocked: $FILE_PATH matches protected pattern '$pattern'" >&2
    exit 2
  fi
done

exit 0

壓縮後重新注入上下文

當 Claude 的上下文視窗填滿時，壓縮會摘要對話以釋放空間。使用帶有 compact 匹配器的 SessionStart hook 在每次壓縮後重新注入關鍵上下文。

{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Reminder: use Bun, not npm. Run bun test before committing. Current sprint: auth refactor.'"
          }
        ]
      }
    ]
  }
}

  }
}

當目錄或檔案更改時重新載入環境

CwdChanged hook 在 Claude 每次更改目錄時執行，因此你可以為新位置重新載入正確的變數。

{
  "hooks": {
    "CwdChanged": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "direnv export bash >> \"$CLAUDE_ENV_FILE\""
          }
        ]
      }
    ]
  }
}

要對特定檔案而不是每個目錄更改做出反應，請使用 FileChanged：

{
  "hooks": {
    "FileChanged": [
      {
        "matcher": ".envrc|.env",
        "hooks": [
          {
            "type": "command",
            "command": "direnv export bash >> \"$CLAUDE_ENV_FILE\""
          }
        ]
      }
    ]
  }
}

自動批准特定權限提示

跳過你總是允許的工具呼叫的批准對話。此範例自動批准 ExitPlanMode：

{
  "hooks": {
    "PermissionRequest": [
      {
        "matcher": "ExitPlanMode",
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"hookSpecificOutput\": {\"hookEventName\": \"PermissionRequest\", \"decision\": {\"behavior\": \"allow\"}}}'"
          }
        ]
      }
    ]
  }
}

保持匹配器盡可能狹窄。匹配 .* 或留下匹配器為空會自動批准每個權限提示。

Hook 事件在 Claude Code 中的特定生命週期點觸發。當事件觸發時，所有匹配的 hooks 並行執行，相同的 hook 指令會自動去重。

每個 hook 都有一個 type 來決定它如何執行：

  • "type": "command"：執行 shell 指令
  • "type": "http"：將事件資料 POST 到 URL
  • "type": "prompt"：單輪 LLM 評估
  • "type": "agent"：具有工具存取權限的多輪驗證

讀取輸入並返回輸出

Hooks 透過 stdin、stdout、stderr 和退出代碼與 Claude Code 通訊。

Hook 輸入：

每個事件都包含常見欄位，如 session_id 和 cwd，但每個事件類型新增不同的資料。例如 PreToolUse hook 在 stdin 上接收：

{
  "session_id": "abc123",
  "cwd": "/Users/sarah/myproject",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "npm test"
  }
}

Hook 輸出：

退出代碼決定接下來會發生什麼：

  • 退出 0：操作繼續。對於 UserPromptSubmit 和 SessionStart hooks，你寫入 stdout 的任何內容都會新增到 Claude 的上下文中。
  • 退出 2：操作被阻止。寫入原因到 stderr，Claude 會收到它作為回饋。
  • 任何其他退出代碼：操作繼續。Stderr 被記錄但不顯示給 Claude。使用 Ctrl+O 切換詳細模式以查看這些訊息。

結構化 JSON 輸出：

退出代碼給你兩個選項：允許或阻止。為了取得更多控制，退出 0 並改為將 JSON 物件印到 stdout。

使用退出 2 以 stderr 訊息阻止，或使用 JSON 退出 0 以取得結構化控制。不要混合它們：Claude Code 在你退出 2 時忽略 JSON。

基於提示的 Hooks

對於需要判斷而不是確定性規則的決策，使用 type: "prompt" hooks。Claude Code 不執行 shell 指令，而是將你的提示和 hook 的輸入資料傳送到 Claude 模型（預設為 Haiku）來做出決策。

模型的唯一工作是返回一個是/否決策作為 JSON：

  • "ok": true：操作繼續
  • "ok": false：操作被阻止。模型的 "reason" 被回饋給 Claude。
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Check if all tasks are complete. If not, respond with {\"ok\": false, \"reason\": \"what remains to be done\"}."
          }
        ]
      }
    ]
  }
}

基於代理的 Hooks

當驗證需要檢查檔案或執行指令時，使用 type: "agent" hooks。與只進行單個 LLM 呼叫的提示 hooks 不同，代理 hooks 產生一個 subagent，它可以讀取檔案、搜尋程式碼和使用其他工具來驗證條件，然後返回決策。

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "agent",
            "prompt": "Verify that all unit tests pass. Run the test suite and check the results. $ARGUMENTS",
            "timeout": 120
          }
        ]
      }
    ]
  }
}

HTTP Hooks

使用 type: "http" hooks 將事件資料 POST 到 HTTP 端點。端點接收指令 hook 在 stdin 上接收的相同 JSON，並使用相同的 JSON 格式透過 HTTP 回應本文返回結果。

{
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [
          {
            "type": "http",
            "url": "http://localhost:8080/hooks/tool-use",
            "headers": {
              "Authorization": "Bearer $MY_TOKEN"
            },
            "allowedEnvVars": ["MY_TOKEN"]
          }
        ]
      }
    ]
  }
}

標頭值支援使用 $VAR_NAME 或 ${VAR_NAME} 語法的環境變數插值。僅解析 allowedEnvVars 陣列中列出的變數。

限制和疑難排解

限制

  • 指令 hooks 僅透過 stdout、stderr 和退出代碼通訊。它們無法直接觸發指令或工具呼叫。
  • Hook 逾時預設為 10 分鐘，可透過 timeout 欄位（以秒為單位）按 hook 設定。
  • PostToolUse hooks 無法撤銷操作，因為工具已經執行。
  • PermissionRequest hooks 不在非互動模式（-p）中觸發。對於自動化權限決策，使用 PreToolUse hooks。
  • Stop hooks 在 Claude 完成回應時觸發，而不僅僅在任務完成時。它們不在使用者中斷時觸發。

Hook 未觸發

  • 執行 /hooks 並確認 hook 出現在正確的事件下
  • 檢查匹配器模式是否與工具名稱完全匹配（匹配器區分大小寫）
  • 驗證你是否觸發了正確的事件類型

Stop Hook 永遠執行

你的 Stop hook 腳本需要檢查它是否已經觸發了繼續。從 JSON 輸入中解析 stop_hook_active 欄位，如果為 true 則提前退出：

#!/bin/bash
INPUT=$(cat)
if [ "$(echo "$INPUT" | jq -r '.stop_hook_active')" = "true" ]; then
  exit 0  # 允許 Claude 停止
fi
# ... 你的 hook 邏輯的其餘部分

JSON 驗證失敗

如果你的 shell 設定檔包含無條件的 echo 語句，該輸出會被新增到你的 hook 的 JSON 前面。要修復此問題：

# 在 ~/.zshrc 或 ~/.bashrc 中
if [[ $- == *i* ]]; then
  echo "Shell ready"
fi

Hooks 是使用者定義的 shell 指令、HTTP 端點或 LLM 提示，在 Claude Code 生命週期中的特定點自動執行。使用此參考查找事件架構、設定選項、JSON 輸入/輸出格式以及非同步 hooks、HTTP hooks 和 MCP 工具 hooks 等進階功能。

Hook 生命週期

Hooks 在 Claude Code 工作階段期間的特定點觸發。當事件觸發且匹配器匹配時，Claude Code 會將關於該事件的 JSON 上下文傳遞給你的 hook 處理程序。對於指令 hooks，輸入透過 stdin 到達。對於 HTTP hooks，它作為 POST 請求本文到達。

設定

Hooks 在 JSON 設定檔中定義。設定有三個巢狀層級：

  1. 選擇要回應的 hook 事件，如 PreToolUse 或 Stop
  2. 新增匹配器群組以過濾何時觸發，如「僅針對 Bash 工具」
  3. 定義一個或多個 hook 處理程序以在匹配時執行
匹配器模式

matcher 欄位是一個正則表達式字串，用於過濾 hooks 何時觸發。使用 "*"、"" 或完全省略 matcher 以匹配所有出現。

匹配器是一個正則表達式，所以 Edit|Write 匹配任一工具，Notebook.* 匹配任何以 Notebook 開頭的工具。

匹配 MCP 工具

MCP 伺服器工具在工具事件中顯示為一般工具，遵循命名模式 mcp__<server>__<tool>，例如：

  • mcp__memory__create_entities：Memory 伺服器的建立實體工具
  • mcp__filesystem__read_file：Filesystem 伺服器的讀取檔案工具

使用正則表達式模式來針對特定 MCP 工具或工具群組：

  • mcp__memory__.* 匹配來自 memory 伺服器的所有工具
  • mcp__.*__write.* 匹配來自任何伺服器的任何包含 "write" 的工具
Hook 處理程序欄位

內部 hooks 陣列中的每個物件都是一個 hook 處理程序。有四種類型：

  • 指令 hooks (type: "command")：執行 shell 指令
  • HTTP hooks (type: "http")：將事件的 JSON 輸入作為 HTTP POST 請求傳送到 URL
  • 提示 hooks (type: "prompt")：向 Claude 模型傳送提示以進行單輪評估
  • 代理 hooks (type: "agent")：產生一個可以使用工具來驗證條件的 subagent

Skills 和代理中的 Hooks

除了設定檔和外掛外，hooks 還可以使用 frontmatter 直接在 skills 和 subagents 中定義。這些 hooks 的範圍限於元件的生命週期，僅在該元件活躍時執行。

/hooks 選單

選單顯示所有四種 hook 類型：command、prompt、agent 和 http。每個 hook 都標有 [type] 前綴和指示其定義位置的來源（User、Project、Local、Plugin、Session、Built-in）。

在背景執行 Hooks

預設情況下，hooks 阻止 Claude 的執行，直到它們完成。對於長時間運行的任務，設定 "async": true 以在背景執行 hook。

非同步 hooks 無法阻止或控制 Claude 的行為。後台程序退出後，如果 hook 產生了帶有 systemMessage 或 additionalContext 欄位的 JSON 回應，該內容在下一個對話回合作為上下文傳遞給 Claude。

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "/path/to/run-tests.sh",
            "async": true,
            "timeout": 120
          }
        ]
      }
    ]
  }
}
非同步 Hooks 的限制
  • 僅 type: "command" hooks 支援 async。基於提示的 hooks 無法非同步執行。
  • 非同步 hooks 無法阻止工具呼叫或返回決策。
  • Hook 輸出在下一個對話回合傳遞。
  • 同一非同步 hook 的多個觸發之間沒有去重。

安全考量

指令 hooks 使用你的系統使用者的完整權限執行。

警告： 指令 hooks 使用你的完整使用者權限執行 shell 指令。它們可以修改、刪除或存取你的使用者帳戶可以存取的任何檔案。在將任何 hook 指令新增到你的設定之前，請審查並測試它們。
安全最佳實務
  • 驗證和清理輸入：永遠不要盲目信任輸入資料
  • 始終引用 shell 變數：使用 "$VAR" 而不是 $VAR
  • 阻止路徑遍歷：檢查檔案路徑中的 ..
  • 使用絕對路徑：為腳本指定完整路徑，使用 "$CLAUDE_PROJECT_DIR" 作為專案根目錄
  • 跳過敏感檔案：避免 .env、.git/、金鑰等
