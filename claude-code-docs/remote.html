遠端控制・頻道・代理

Remote Control 遠端操作、Channels 事件推播、子代理、Agent Teams 團隊協作——當一個 Claude 不夠用的時候。

使用 Remote Control 從任何裝置繼續本地工作階段

使用 Remote Control 從手機、平板電腦或任何瀏覽器繼續本地 Claude Code 工作階段。適用於 claude.ai/code 和 Claude 行動應用程式。

Remote Control 在所有方案中都可用。在 Team 和 Enterprise 上，管理員必須先在 Claude Code 管理員設定中啟用 Remote Control 開關，否則預設為關閉。

Remote Control 將 claude.ai/code 或 Claude 應用程式（iOS 和 Android）連接到在你的機器上執行的 Claude Code 工作階段。在辦公桌上啟動一個任務，然後從沙發上的手機或另一台電腦上的瀏覽器繼續。

當你在機器上啟動 Remote Control 工作階段時，Claude 始終在本地執行，因此沒有任何內容移動到雲端。使用 Remote Control，你可以：

  • 遠端使用完整的本地環境：檔案系統、MCP 伺服器、工具和專案設定都保持可用
  • 同時從兩個介面工作：對話在所有連線裝置上保持同步，因此你可以從終端、瀏覽器和手機交替傳送訊息
  • 在中斷後恢復：如果筆電進入睡眠或網路斷線，當機器重新上線時，工作階段會自動重新連線

與網頁版 Claude Code（在雲端基礎設施上執行）不同，Remote Control 工作階段直接在你的機器上執行並與本地檔案系統互動。網頁和行動介面只是該本地工作階段的一個視窗。

Remote Control 需要 Claude Code v2.1.51 或更高版本。使用 claude --version 檢查版本。

本頁涵蓋設定、如何啟動和連接工作階段，以及 Remote Control 與網頁版 Claude Code 的比較。

需求

在使用 Remote Control 之前，請確認環境滿足以下條件：

  • 訂閱：在 Pro、Max、Team 和 Enterprise 方案中可用。不支援 API 金鑰。在 Team 和 Enterprise 上，管理員必須先在 Claude Code 管理員設定中啟用 Remote Control 開關。
  • 驗證：執行 claude 並使用 /login 透過 claude.ai 登入（如果尚未登入）。
  • 工作區信任：在專案目錄中至少執行一次 claude 以接受工作區信任對話框。

啟動 Remote Control 工作階段

你可以啟動專用的 Remote Control 伺服器、啟動啟用了 Remote Control 的互動式工作階段，或連接已在執行的工作階段。

伺服器模式

導覽到專案目錄並執行：

claude remote-control

該程序在終端中以伺服器模式持續執行，等待遠端連線。它會顯示一個工作階段 URL，你可以使用該 URL 從另一個裝置連接，也可以按空白鍵顯示 QR 碼以從手機快速存取。當遠端工作階段啟用時，終端會顯示連線狀態和工具活動。

可用旗標：

旗標 說明
--name "My Project" 設定自訂工作階段標題，在 claude.ai/code 的工作階段列表中可見。
--spawn <mode> 如何建立並行工作階段。執行時按 w 切換。
• same-dir（預設）：所有工作階段共享目前工作目錄，如果編輯相同檔案可能會衝突。
• worktree：每個按需工作階段都取得自己的 git worktree。需要 git 儲存庫。
--capacity <N> 最大並行工作階段數。預設為 32。
--verbose 顯示詳細的連線和工作階段日誌。
--sandbox / --no-sandbox 啟用或停用沙箱以進行檔案系統和網路隔離。預設關閉。

互動式工作階段

要啟動啟用了 Remote Control 的普通互動式 Claude Code 工作階段，使用 --remote-control 旗標（或 --rc）：

claude --remote-control

可選地為工作階段傳遞一個名稱：

claude --remote-control "My Project"

這讓你在終端中擁有完整的互動式工作階段，同時也可以從 claude.ai 或 Claude 應用程式控制。與 claude remote-control（伺服器模式）不同，你可以在工作階段也可遠端使用時在本地輸入訊息。

從現有工作階段

如果你已在 Claude Code 工作階段中並想遠端繼續它，使用 /remote-control（或 /rc）命令：

/remote-control

傳遞名稱作為參數以設定自訂工作階段標題：

/remote-control My Project

這會啟動一個 Remote Control 工作階段，繼承目前的對話歷史記錄，並顯示一個工作階段 URL 和 QR 碼，你可以使用它從另一個裝置連接。--verbose、--sandbox 和 --no-sandbox 旗標不適用於此命令。

從另一個裝置連接

一旦 Remote Control 工作階段啟用，你有幾種方式從另一個裝置連接：

  • 開啟工作階段 URL：在任何瀏覽器中直接前往 claude.ai/code 上的工作階段。claude remote-control 和 /remote-control 都在終端中顯示此 URL。
  • 掃描 QR 碼：顯示在工作階段 URL 旁邊，直接在 Claude 應用程式中開啟。使用 claude remote-control 時，按空白鍵切換 QR 碼顯示。
  • 開啟 claude.ai/code 或 Claude 應用程式：在工作階段列表中按名稱尋找工作階段。Remote Control 工作階段在線上時顯示帶有綠色狀態點的電腦圖示。

遠端工作階段標題按以下順序選擇：

  1. 你傳遞給 --name、--remote-control 或 /remote-control 的名稱
  2. 你使用 /rename 設定的標題
  3. 現有對話歷史記錄中的最後一條有意義的訊息
  4. 你傳送的第一條提示

如果環境已有啟用的工作階段，你會被詢問是否繼續它或啟動新工作階段。

如果你還沒有 Claude 應用程式，在 Claude Code 中使用 /mobile 命令顯示 iOS 或 Android 的下載 QR 碼。

為所有工作階段啟用 Remote Control

預設情況下，Remote Control 僅在你顯式執行 claude remote-control、claude --remote-control 或 /remote-control 時啟用。要為每個互動式工作階段自動啟用它，在 Claude Code 中執行 /config 並將**為所有工作階段啟用 Remote Control** 設定為 true。設回 false 以停用。

啟用此設定後，每個互動式 Claude Code 程序註冊一個遠端工作階段。如果你執行多個實例，每個實例都取得自己的環境和工作階段。要從單個程序執行多個並行工作階段，請改用帶有 --spawn 的伺服器模式。

連線和安全

你的本地 Claude Code 工作階段僅發出出站 HTTPS 請求，從不在機器上開啟入站埠。當你啟動 Remote Control 時，它向 Anthropic API 註冊並輪詢工作。當你從另一個裝置連接時，伺服器透過串流連線在網頁或行動用戶端和本地工作階段之間路由訊息。

所有流量都透過 Anthropic API 以 TLS 傳輸，與任何 Claude Code 工作階段的傳輸安全相同。連線使用多個短期憑證，每個憑證的範圍限定為單一用途並獨立過期。

Remote Control 與網頁版 Claude Code 的比較

Remote Control 和網頁版 Claude Code 都使用 claude.ai/code 介面。關鍵區別在於工作階段執行的位置：Remote Control 在你的機器上執行，因此本地 MCP 伺服器、工具和專案設定保持可用。網頁版 Claude Code 在 Anthropic 管理的雲端基礎設施中執行。

當你正在進行本地工作並想從另一個裝置繼續時，使用 Remote Control。當你想在沒有任何本地設定的情況下啟動任務、處理你沒有 clone 的儲存庫或並行執行多個任務時，使用網頁版 Claude Code。

限制

  • 每個互動式程序一個遠端工作階段：在伺服器模式之外，每個 Claude Code 實例一次支援一個遠端工作階段。使用帶有 --spawn 的伺服器模式從單個程序執行多個並行工作階段。
  • 終端必須保持開啟：Remote Control 作為本地程序執行。如果你關閉終端或停止 claude 程序，工作階段結束。再次執行 claude remote-control 以啟動新工作階段。
  • 延長的網路中斷：如果機器處於喚醒狀態但無法在約 10 分鐘以上到達網路，工作階段逾時並且程序退出。再次執行 claude remote-control 以啟動新工作階段。

疑難排解

"Remote Control 尚未為你的帳號啟用"

在存在某些環境變數的情況下，資格檢查可能會失敗：

  • CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC 或 DISABLE_TELEMETRY：取消設定它們並重試。
  • CLAUDE_CODE_USE_BEDROCK、CLAUDE_CODE_USE_VERTEX 或 CLAUDE_CODE_USE_FOUNDRY：Remote Control 需要 claude.ai 驗證，不適用於第三方供應商。

如果這些都沒有設定，執行 /logout 然後 /login 以重新整理。

"Remote Control 被你的組織的政策停用"

此錯誤有三個不同的原因。先執行 /status 以查看你使用的登入方法和訂閱。

  • 你使用 API 金鑰或 Console 帳號進行驗證：Remote Control 需要 claude.ai OAuth。執行 /login 並選擇 claude.ai 選項。如果環境中設定了 ANTHROPIC_API_KEY，請取消設定它。
  • 你的 Team 或 Enterprise 管理員尚未啟用它：Remote Control 在這些方案上預設為關閉。管理員可以在 claude.ai/admin-settings/claude-code 開啟 **Remote Control** 開關來啟用它。這是伺服器端組織設定，不是僅管理設定金鑰。
  • 管理員開關呈灰色：你的組織有資料保留或合規設定與 Remote Control 不相容。這無法從管理面板更改。請聯絡 Anthropic 支援以討論選項。

"Remote credentials fetch failed"

Claude Code 無法從 Anthropic API 取得短期憑證以建立連線。使用 --verbose 重新執行以查看完整錯誤：

claude remote-control --verbose

常見原因：

  • 未登入：執行 claude 並使用 /login 以你的 claude.ai 帳號進行驗證。Remote Control 不支援 API 金鑰驗證。
  • 網路或代理問題：防火牆或代理可能阻止出站 HTTPS 請求。Remote Control 需要存取埠 443 上的 Anthropic API。
  • 工作階段建立失敗：如果你還看到 Session creation failed — see debug log，失敗發生在設定的早期。檢查你的訂閱是否處於啟用狀態。

使用 Channels 將事件推播到執行中的工作階段

使用 channels 從 MCP 伺服器將訊息、警報和 webhooks 推播到 Claude Code 工作階段中。轉發 CI 結果、聊天訊息和監控事件，以便 Claude 在你離開時做出反應。

Channels 處於研究預覽階段，需要 Claude Code v2.1.80 或更高版本。需要 claude.ai 登入。不支援主控台和 API 金鑰驗證。Team 和 Enterprise 組織必須明確啟用它們。

Channel 是一個 MCP 伺服器，它將事件推播到你執行中的 Claude Code 工作階段中，以便 Claude 可以對你不在終端時發生的事情做出反應。Channels 可以是雙向的：Claude 讀取事件並透過同一 channel 回覆，就像聊天橋接一樣。事件僅在工作階段開啟時到達，因此對於始終在線的設定，你可以在背景程序或持久終端中執行 Claude。

與產生新的雲端工作階段或等待被輪詢的整合不同，事件到達你已經開啟的工作階段中。

你將 channel 作為外掛安裝並使用自己的憑證設定它。Telegram、Discord 和 iMessage 包含在研究預覽中。

當 Claude 透過 channel 回覆時，你會在終端中看到入站訊息，但看不到回覆文字。終端顯示工具呼叫和確認（如「已傳送」），實際回覆出現在其他平台上。

支援的 Channels

每個支援的 channel 都是一個需要 Bun 的外掛。在連接真實平台之前，要取得外掛流程的實際示範，嘗試 fakechat 快速入門。

Telegram

查看完整的 Telegram 外掛原始碼。

步驟 1：建立 Telegram 機器人

在 Telegram 中開啟 BotFather 並傳送 /newbot。給它一個顯示名稱和一個以 bot 結尾的唯一使用者名稱。複製 BotFather 回傳的權杖（token）。

步驟 2：安裝外掛

在 Claude Code 中，執行：

/plugin install telegram@claude-plugins-official

如果 Claude Code 回報在任何市集中都找不到該外掛，你的市集可能遺失或已過期。執行 /plugin marketplace update claude-plugins-official 來重新整理它，或者如果你之前沒有加過，執行 /plugin marketplace add anthropics/claude-plugins-official。然後重試安裝。

安裝後，執行 /reload-plugins 來啟用外掛的設定命令。

步驟 3：設定權杖

使用來自 BotFather 的權杖執行設定命令：

/telegram:configure <token>

這會將其儲存到 ~/.claude/channels/telegram/.env。你也可以在啟動 Claude Code 之前在 shell 環境中設定 TELEGRAM_BOT_TOKEN。

步驟 4：重啟並啟用 channels

退出 Claude Code 並使用 channel 旗標重啟。這會啟動 Telegram 外掛，它開始輪詢來自機器人的訊息：

claude --channels plugin:telegram@claude-plugins-official

步驟 5：配對帳號

開啟 Telegram 並向機器人傳送任何訊息。機器人會回覆一個配對碼。

如果機器人沒有回應，請確保 Claude Code 正在使用上一步中的 --channels 執行。機器人只能在 channel 啟用時回覆。

回到 Claude Code，執行：

/telegram:access pair <code>

然後鎖定存取權限，以便只有你的帳號可以傳送訊息：

/telegram:access policy allowlist

Discord

查看完整的 Discord 外掛原始碼。

步驟 1：建立 Discord 機器人

前往 Discord 開發者入口，點選 **New Application**（新應用程式），並為其命名。在 **Bot**（機器人）部分中，建立使用者名稱，然後點選 **Reset Token**（重設權杖）並複製權杖。

步驟 2：啟用消息內容意圖

在機器人設定中，捲動到 **Privileged Gateway Intents**（特權閘道意圖）並啟用 **Message Content Intent**（消息內容意圖）。

步驟 3：邀請機器人加入伺服器

前往 **OAuth2 > URL Generator**。選擇 bot 範圍並啟用這些權限：

  • View Channels（檢視頻道）
  • Send Messages（傳送訊息）
  • Send Messages in Threads（在討論串中傳送訊息）
  • Read Message History（讀取訊息歷史）
  • Attach Files（附加檔案）
  • Add Reactions（新增反應）

開啟產生的 URL 以將機器人加入你的伺服器。

步驟 4：安裝外掛

在 Claude Code 中，執行：

/plugin install discord@claude-plugins-official

安裝後，執行 /reload-plugins 來啟用外掛的設定命令。

步驟 5：設定權杖

使用複製的機器人權杖執行設定命令：

/discord:configure <token>

這會將其儲存到 ~/.claude/channels/discord/.env。你也可以在啟動 Claude Code 之前在 shell 環境中設定 DISCORD_BOT_TOKEN。

步驟 6：重啟並啟用 channels

退出 Claude Code 並使用 channel 旗標重啟：

claude --channels plugin:discord@claude-plugins-official

步驟 7：配對帳號

在 Discord 上向機器人傳送私訊。機器人會回覆一個配對碼。

回到 Claude Code，執行：

/discord:access pair <code>

然後鎖定存取權限：

/discord:access policy allowlist

iMessage

查看完整的 iMessage 外掛原始碼。

iMessage channel 直接讀取你的訊息資料庫，並透過 AppleScript 傳送回覆。它需要 macOS，不需要機器人權杖或外部服務。

步驟 1：授予完整磁碟存取權限

位於 ~/Library/Messages/chat.db 的訊息資料庫受 macOS 保護。伺服器第一次讀取它時，macOS 會提示存取權限。

步驟 2：安裝外掛

/plugin install imessage@claude-plugins-official

步驟 3：重啟並啟用 channels

claude --channels plugin:imessage@claude-plugins-official

步驟 4：給自己傳簡訊

在任何登入到你的 Apple ID 的裝置上開啟訊息，並向自己傳送訊息。它立即到達 Claude：自我聊天繞過存取控制，無需設定。

Claude 傳送的第一條回覆會觸發 macOS 自動化提示，詢問你的終端是否可以控制訊息。點選 **OK**（確定）。

步驟 5：允許其他傳送者

預設情況下，只有你自己的訊息通過。要讓另一個聯絡人到達 Claude，加入他們的控制代碼（handle）：

/imessage:access allow +15551234567

控制代碼是 +country 格式的電話號碼或 Apple ID 電子郵件，如 user@example.com。

你也可以建立自己的 channel，用於尚未有外掛的系統。

快速入門

Fakechat 是一個官方支援的示範 channel，在 localhost 上執行聊天 UI，無需驗證，也無需設定外部服務。

安裝並啟用 fakechat 後，你可以在瀏覽器中輸入，訊息會到達 Claude Code 工作階段。Claude 回覆，回覆會顯示在瀏覽器中。

要嘗試 fakechat 示範，你需要：

  • Claude Code 已安裝並使用 claude.ai 帳號進行驗證
  • Bun 已安裝。預建的 channel 外掛是 Bun 指令碼。使用 bun --version 檢查；如果失敗，安裝 Bun。
  • Team/Enterprise 使用者：你的組織管理員必須在託管設定中啟用 channels

步驟 1：安裝 fakechat channel 外掛

啟動 Claude Code 工作階段並執行安裝命令：

/plugin install fakechat@claude-plugins-official

步驟 2：重啟並啟用 channel

退出 Claude Code，然後使用 --channels 重啟並傳遞你安裝的 fakechat 外掛：

claude --channels plugin:fakechat@claude-plugins-official

fakechat 伺服器會自動啟動。

提示：你可以將多個外掛傳遞給 --channels，用空格分隔。

步驟 3：推播訊息進來

在 http://localhost:8787 開啟 fakechat UI 並輸入訊息：

hey, what's in my working directory?

訊息作為 <channel source="fakechat"> 事件到達 Claude Code 工作階段。Claude 讀取它，完成工作，並呼叫 fakechat 的 reply 工具。答案顯示在聊天 UI 中。

如果 Claude 在你離開終端時遇到權限提示，工作階段會暫停直到你回應。宣告權限中繼功能的 Channel 伺服器可以將這些提示轉發給你，以便你可以遠端批准或拒絕。對於無人值守使用，--dangerously-skip-permissions 完全繞過提示，但僅在你信任的環境中使用。

安全性

每個已批准的 channel 外掛都維護一個傳送者允許清單：只有你加入的 ID 可以推播訊息，其他所有人都會被靜默丟棄。

Telegram 和 Discord 透過配對來引導清單：

  1. 在 Telegram 或 Discord 中找到機器人並向其傳送任何訊息
  2. 機器人回覆一個配對碼
  3. 在 Claude Code 工作階段中，在提示時批准代碼
  4. 你的傳送者 ID 被加入到允許清單

iMessage 的工作方式不同：給自己傳簡訊會自動繞過門禁，你可以使用 /imessage:access allow 透過控制代碼加入其他聯絡人。

除此之外，你可以使用 --channels 控制每個工作階段啟用哪些伺服器，在 Team 和 Enterprise 方案上，你的組織可以使用 channelsEnabled 控制可用性。

僅在 .mcp.json 中還不足以推播訊息：伺服器還必須在 --channels 中指名。

允許清單也會限制權限中繼（如果 channel 宣告了它）。任何可以透過 channel 回覆的人都可以批准或拒絕你工作階段中的工具使用，因此只將你信任具有該權限的傳送者加入允許清單。

Enterprise 控制

在 Team 和 Enterprise 方案上，channels 預設關閉。管理員透過兩個託管設定控制可用性，使用者無法覆蓋：

設定 用途 未設定時
channelsEnabled 主開關。必須為 true 才能讓任何 channel 傳遞訊息。透過 claude.ai 管理員主控台切換或直接在託管設定中設定。關閉時阻止所有 channels，包括開發旗標。 Channels 被阻止
allowedChannelPlugins 啟用 channels 後哪些外掛可以註冊。設定時取代 Anthropic 維護的清單。僅在 channelsEnabled 為 true 時適用。 套用 Anthropic 預設清單

沒有組織的 Pro 和 Max 使用者完全跳過這些檢查：channels 可用，使用者使用 --channels 按工作階段選擇加入。

為你的組織啟用 channels

管理員可以從 claude.ai > Admin Settings > Claude Code > Channels 啟用 channels，或透過在託管設定中將 channelsEnabled 設定為 true。

啟用後，組織中的使用者可以使用 --channels 將 channel 伺服器選擇加入到各個工作階段中。如果設定被停用或未設定，MCP 伺服器仍會連線，其工具可以工作，但 channel 訊息不會到達。啟動警告會告訴使用者讓管理員啟用該設定。

限制哪些 channel 外掛可以執行

預設情況下，Anthropic 維護的允許清單上的任何外掛都可以註冊為 channel。Team 和 Enterprise 方案上的管理員可以透過在託管設定中設定 allowedChannelPlugins 來取代該允許清單：

{
  "channelsEnabled": true,
  "allowedChannelPlugins": [
    { "marketplace": "claude-plugins-official", "plugin": "telegram" },
    { "marketplace": "claude-plugins-official", "plugin": "discord" },
    { "marketplace": "acme-corp-plugins", "plugin": "internal-alerts" }
  ]
}

設定 allowedChannelPlugins 時，它完全取代 Anthropic 允許清單：只有列出的外掛可以註冊。保持未設定以回退到預設 Anthropic 允許清單。空陣列阻止所有 channel 外掛從允許清單中，但 --dangerously-load-development-channels 仍可以為本地測試繞過它。要完全阻止 channels（包括開發旗標），請保持 channelsEnabled 未設定。

研究預覽

Channels 是一個研究預覽功能。可用性正在逐步推出，--channels 旗標語法和協定合約可能會根據回饋而改變。

在預覽期間，--channels 僅接受來自 Anthropic 維護的允許清單的外掛，或來自你組織的允許清單（如果管理員已設定 allowedChannelPlugins）。claude-plugins-official 中的 channel 外掛是預設批准的集合。

要測試你正在建立的 channel，使用 --dangerously-load-development-channels。

在 Claude Code GitHub 儲存庫上回報問題或回饋。

Channel 是一個 MCP 伺服器，它將事件推播到 Claude Code 工作階段中，以便 Claude 可以對終端外發生的事情做出反應。

你可以建立單向或雙向頻道。單向頻道轉發警報、webhooks 或監控事件供 Claude 處理。雙向頻道（如聊天橋接）也公開回覆工具，以便 Claude 可以傳送訊息回覆。具有受信任傳送者路徑的頻道也可以選擇加入中繼權限提示，以便你可以遠端批准或拒絕工具使用。

  • 概述：頻道如何工作
  • 你需要什麼：需求和一般步驟
  • 範例：建立 webhook 接收器：最小單向演練
  • 伺服器選項：建構函式欄位
  • 通知格式：事件負載
  • 公開回覆工具：讓 Claude 傳送訊息回覆
  • 閘門控制入站訊息：傳送者檢查以防止提示注入
  • 中繼權限提示：將工具批准提示轉發到遠端頻道

要使用現有頻道而不是建立一個，參見 Channels。Telegram、Discord、iMessage 和 fakechat 包含在研究預覽中。

概述

Channel 是一個在與 Claude Code 相同的機器上執行的 MCP 伺服器。Claude Code 將其作為子程序產生並透過 stdio 進行通訊。你的頻道伺服器是外部系統和 Claude Code 工作階段之間的橋樑：

  • 聊天平台（Telegram、Discord）：你的外掛在本地執行並輪詢平台的 API 以取得新訊息。當有人向機器人傳送 DM 時，外掛接收訊息並將其轉發給 Claude。無需公開 URL。
  • Webhooks（CI、監控）：你的伺服器在本地 HTTP 埠上監聽。外部系統 POST 到該埠，你的伺服器將負載推播到 Claude。

你需要什麼

唯一的硬性需求是 @modelcontextprotocol/sdk 套件和 Node.js 相容的執行環境。Bun、Node 和 Deno 都可以。

你的伺服器需要：

  1. 宣告 claude/channel 能力，以便 Claude Code 註冊通知監聽器
  2. 當發生某事時發出 notifications/claude/channel 事件
  3. 透過 stdio transport 連接（Claude Code 將你的伺服器作為子程序產生）

範例：建立 Webhook 接收器

本演練建立一個單檔伺服器，該伺服器監聽 HTTP 請求並將其轉發到 Claude Code 工作階段中。最後，任何可以傳送 HTTP POST 的東西（如 CI 管線、監控警報或 curl 命令）都可以將事件推播到 Claude。

步驟 1：建立專案

mkdir webhook-channel && cd webhook-channel
bun add @modelcontextprotocol/sdk

步驟 2：撰寫頻道伺服器

建立一個名為 webhook.ts 的檔案：

#!/usr/bin/env bun
import { Server } from '@modelcontextprotocol/sdk/server/index.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'

// 建立 MCP 伺服器並將其宣告為頻道
const mcp = new Server(
  { name: 'webhook', version: '0.0.1' },
  {
    // 這個鍵使其成為頻道 — Claude Code 為其註冊監聽器
    capabilities: { experimental: { 'claude/channel': {} } },
    // 加入到 Claude 的系統提示，以便它知道如何處理這些事件
    instructions: 'Events from the webhook channel arrive as <channel source="webhook" ...>. They are one-way: read them and act, no reply expected.',
  },
)

// 透過 stdio 連接到 Claude Code（Claude Code 產生此程序）
await mcp.connect(new StdioServerTransport())

// 啟動一個 HTTP 伺服器，將每個 POST 轉發給 Claude
Bun.serve({
  port: 8788,
  hostname: '127.0.0.1',
  async fetch(req) {
    const body = await req.text()
    await mcp.notification({
      method: 'notifications/claude/channel',
      params: {
        content: body,
        meta: { path: new URL(req.url).pathname, method: req.method },
      },
    })
    return new Response('ok')
  },
})

步驟 3：向 Claude Code 註冊你的伺服器

{
  "mcpServers": {
    "webhook": { "command": "bun", "args": ["./webhook.ts"] }
  }
}

步驟 4：測試它

claude --dangerously-load-development-channels server:webhook

在另一個終端中：

curl -X POST localhost:8788 -d "build failed on main: https://ci.example.com/run/1234"

負載作為 <channel> 標籤到達 Claude Code 工作階段中：

<channel source="webhook" path="/" method="POST">build failed on main: https://ci.example.com/run/1234</channel>

在研究預覽期間測試

在研究預覽期間，自訂頻道不在批准的允許清單上。開發旗標在確認提示後繞過特定條目的允許清單：

# 測試你正在開發的外掛
claude --dangerously-load-development-channels plugin:yourplugin@yourmarketplace

# 測試裸 .mcp.json 伺服器（尚無外掛包裝器）
claude --dangerously-load-development-channels server:webhook

此旗標僅跳過允許清單。channelsEnabled 組織政策仍然適用。不要使用它來執行來自不受信任來源的頻道。

伺服器選項

頻道在 Server 建構函式中設定這些選項：

欄位 型別 說明
capabilities.experimental['claude/channel'] object 必要。始終為 {}。存在即註冊通知監聽器。
capabilities.experimental['claude/channel/permission'] object 選用。始終為 {}。宣告此頻道可以接收權限中繼請求。參見中繼權限提示。
capabilities.tools object 僅雙向。始終為 {}。標準 MCP 工具能力。參見公開回覆工具。
instructions string 建議。加入到 Claude 的系統提示。告訴 Claude 期望什麼事件、<channel> 標籤屬性的含義、是否回覆，如果是，使用哪個工具以及傳回哪個屬性（如 chat_id）。

通知格式

你的伺服器使用兩個參數發出 notifications/claude/channel：

欄位 型別 說明
content string 事件主體。作為 <channel> 標籤的主體傳遞。
meta Record<string, string> 選用。每個條目成為 <channel> 標籤上的屬性，用於路由上下文，如聊天 ID、傳送者名稱或警報嚴重性。鍵必須是識別碼：僅字母、數字和底線。

範例：

await mcp.notification({
  method: 'notifications/claude/channel',
  params: {
    content: 'build failed on main: https://ci.example.com/run/1234',
    meta: { severity: 'high', run_id: '1234' },
  },
})

事件在 Claude 的上下文中到達：

<channel source="your-channel" severity="high" run_id="1234">
build failed on main: https://ci.example.com/run/1234
</channel>

公開回覆工具

如果頻道是雙向的（如聊天橋接），公開一個標準 MCP 工具，Claude 可以呼叫它來傳送訊息回覆。回覆工具有三個元件：

  1. Server 建構函式能力中的 tools: {} 條目
  2. 定義工具架構並實作傳送邏輯的工具處理常式
  3. Server 建構函式中的 instructions 字串，告訴 Claude 何時以及如何呼叫工具

閘門控制入站訊息

未閘門控制的頻道是提示注入向量。任何可以到達你的端點的人都可以在 Claude 前面放置文字。在呼叫 mcp.notification() 之前，根據允許清單檢查傳送者：

const allowed = new Set(loadAllowlist())

if (!allowed.has(message.from.id)) {  // 依傳送者，不是房間
  return  // 靜默丟棄
}
await mcp.notification({ ... })

根據傳送者的身分（message.from.id）而不是聊天或房間 ID（message.chat.id）進行閘門控制。在群組聊天中，這些不同，根據房間進行閘門控制會讓任何人向工作階段注入訊息。

中繼權限提示

權限中繼需要 Claude Code v2.1.81 或更高版本。

當 Claude 呼叫需要批准的工具時，本地終端對話開啟，工作階段等待。雙向頻道可以選擇加入以在平行接收相同的提示，並將其中繼到你的另一台裝置。兩者都保持啟用：你可以在終端或手機上回答，Claude Code 套用先到達的任何答案並關閉另一個。

中繼如何工作

當權限提示開啟時，中繼循環有四個步驟：

  1. Claude Code 產生一個短請求 ID 並通知你的伺服器
  2. 你的伺服器將提示和 ID 轉發到聊天應用程式
  3. 遠端使用者使用該 ID 回覆是或否
  4. 你的入站處理常式將回覆解析為判決，Claude Code 僅在 ID 匹配開放請求時套用它

權限請求欄位

來自 Claude Code 的出站通知是 notifications/claude/channel/permission_request：

欄位 說明
request_id 從 a-z 中抽取的五個小寫字母，不包括 l。將其包含在出站提示中以便在回覆中回傳。
tool_name Claude 想要使用的工具名稱，例如 Bash 或 Write。
description 此特定工具呼叫的人類可讀摘要。
input_preview 工具的參數作為 JSON 字串，截斷為 200 個字元。

你的伺服器傳送回的判決是 notifications/claude/channel/permission，有兩個欄位：request_id 回傳上面的 ID，behavior 設定為 'allow' 或 'deny'。

向聊天橋接加入中繼

需要三個元件：

  1. Server 建構函式中 experimental 能力下的 claude/channel/permission: {} 條目
  2. notifications/claude/channel/permission_request 的通知處理常式
  3. 入站訊息處理常式中識別 yes <id> 或 no <id> 的檢查

建立自訂子代理

在 Claude Code 中建立和使用專門的 AI 子代理，用於特定任務的工作流程和改進的上下文管理。

Subagents（子代理）是處理特定類型任務的專門 AI 助手。每個 subagent 在自己的 context window（上下文視窗）中執行，具有自訂系統提示、特定的工具存取權限和獨立的權限。當 Claude 遇到與 subagent 描述相匹配的任務時，它會委託給該 subagent，該 subagent 獨立工作並回傳結果。

如果你需要多個代理並行工作並相互通訊，參見 agent teams。Subagents 在單個工作階段中工作；agent teams 跨多個工作階段進行協調。

Subagents 幫助你：

  • 保留上下文，通過將探索和實作保持在主對話之外
  • 強制執行約束，通過限制 subagent 可以使用的工具
  • 跨專案重用設定，使用使用者級 subagents
  • 專門化行為，為特定領域使用專注的系統提示
  • 控制成本，通過將任務路由到更快、更便宜的模型（如 Haiku）

Claude 使用每個 subagent 的描述來決定何時委託任務。建立 subagent 時，請撰寫清晰的描述，以便 Claude 知道何時使用它。

內建子代理

Claude Code 包括幾個內建 subagents，Claude 在適當時自動使用：

Explore

一個快速的、唯讀的代理，針對搜尋和分析程式碼庫進行了最佳化。

  • Model（模型）: Haiku（快速、低延遲）
  • Tools（工具）: 唯讀工具（拒絕存取 Write 和 Edit 工具）
  • Purpose（用途）: 檔案發現、程式碼搜尋、程式碼庫探索

當 Claude 需要搜尋或理解程式碼庫而不進行更改時，它會委託給 Explore。呼叫時，Claude 指定彻底程度級別：**quick** 用於有針對性的查找，**medium** 用於平衡的探索，或 **very thorough** 用於全面分析。

Plan

一個研究代理，在 plan mode 期間使用，在呈現計畫之前收集上下文。

  • Model: 從主對話繼承
  • Tools: 唯讀工具
  • Purpose: 用於規劃的程式碼庫研究

General-purpose（通用）

一個能夠處理複雜、多步驟任務的代理。

  • Model: 從主對話繼承
  • Tools: 所有工具
  • Purpose: 複雜研究、多步驟操作、程式碼修改

快速入門：建立你的第一個 Subagent

Subagents 在帶有 YAML frontmatter 的 Markdown 檔案中定義。你可以手動建立它們或使用 /agents 命令。

設定 Subagents

使用 /agents 命令

/agents 命令提供互動式介面來管理 subagents。要從命令列列出所有已設定的 subagents，執行 claude agents。

選擇 Subagent 範圍

位置 範圍 優先順序 如何建立

CLI 定義的 subagents 範例：

撰寫 Subagent 檔案

Subagent 檔案使用 YAML frontmatter 進行設定，然後是 Markdown 中的系統提示：

---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: sonnet
---

You are a code reviewer. When invoked, analyze the code and provide
specific, actionable feedback on quality, security, and best practices.

支援的 frontmatter 欄位

欄位 必要 說明
name 是 使用小寫字母和連字號的唯一識別碼
description 是 Claude 何時應該委託給此 subagent
tools 否 subagent 可以使用的工具。省略則繼承所有工具
disallowedTools 否 要拒絕的工具，從繼承或指定的清單中移除
model 否 使用的模型：sonnet、opus、haiku、完整模型 ID 或 inherit。預設為 inherit
permissionMode 否 權限模式：default、acceptEdits、dontAsk、bypassPermissions 或 plan
maxTurns 否 subagent 停止前的最大代理輪數
skills 否 在啟動時載入到 subagent 上下文中的技能
mcpServers 否 對此 subagent 可用的 MCP 伺服器
hooks 否 限定於此 subagent 的生命週期鉤子
memory 否 持久記憶範圍：user、project 或 local
background 否 設為 true 以始終在背景執行此 subagent
effort 否 此 subagent 啟用時的努力級別。選項：low、medium、high、max（僅 Opus 4.6）
isolation 否 設為 worktree 以在臨時 git worktree 中執行 subagent
initialPrompt 否 當此代理作為主工作階段代理執行時，自動提交為第一個使用者輪次

選擇模型

model 欄位控制 subagent 使用的 AI 模型：

  • Model alias（模型別名）: sonnet、opus 或 haiku
  • Full model ID（完整模型 ID）: 如 claude-opus-4-6 或 claude-sonnet-4-6
  • inherit（繼承）: 使用與主對話相同的模型
  • 省略: 預設為 inherit

解析優先順序：

  1. CLAUDE_CODE_SUBAGENT_MODEL 環境變數
  2. 每次呼叫的 model 參數
  3. Subagent 定義的 model frontmatter
  4. 主對話的模型

可用工具

使用 tools 欄位（允許清單）或 disallowedTools 欄位（拒絕清單）來限制工具。如果兩者都設定，disallowedTools 先套用。

限制可以產生哪些 Subagents

使用 Subagents

理解自動委託

Claude 根據任務描述、subagent 設定中的 description 欄位和目前上下文自動委託任務。

顯式呼叫 Subagents

三種模式：

  • 自然語言：在提示中命名 subagent；Claude 決定是否委託
  • @-mention：保證 subagent 為一個任務執行
  • 工作階段範圍：整個工作階段使用該 subagent 的系統提示、工具限制和模型
claude --agent code-reviewer

要使其成為專案中每個工作階段的預設值：

{
  "agent": "code-reviewer"
}

在前台或背景執行 Subagents

  • 前台 subagents 阻塞主對話直到完成
  • 背景 subagents 在你繼續工作時並行執行

按 **Ctrl+B** 將執行中的任務放在背景。

常見模式

  • 隔離高容量操作：委託產生大量輸出的操作（如測試執行）給 subagent
  • 執行並行研究：產生多個 subagents 同時工作
  • 鏈接 subagents：按順序使用 subagents

在 Subagents 和主對話之間選擇

使用**主對話**的情況：任務需要頻繁來回、多階段共享上下文、快速有針對性的更改、延遲很重要。

使用**subagents**的情況：任務產生大量詳細輸出、想強制執行特定的工具限制或權限、工作是自包含的。

管理 Subagent 上下文

恢復 Subagents

要繼續現有 subagent 的工作而不是重新開始，要求 Claude 恢復它。恢復的 subagents 保留完整的對話歷史。

自動壓縮

Subagents 支援使用與主對話相同的邏輯進行自動壓縮。預設在約 95% 容量時觸發。

協調 Claude Code 工作階段團隊

協調多個 Claude Code 實例作為一個團隊一起工作，具有共享任務、代理間訊息傳遞和集中管理。

Agent teams 是實驗性功能，預設停用。通過將 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS 加入到 settings.json 或環境變數來啟用。Agent teams 在已知限制中存在關於工作階段恢復、任務協調和關閉行為的問題。

Agent teams 讓你協調多個 Claude Code 實例一起工作。一個工作階段充當團隊負責人（team lead），協調工作、分配任務和綜合結果。隊友（teammates）獨立工作，每個都在自己的 context window 中，並直接相互通訊。

與 subagents 不同（subagents 在單個工作階段中執行，只能向主代理回報），你也可以直接與個別隊友互動，無需透過負責人。

Agent teams 需要 Claude Code v2.1.32 或更高版本。

何時使用 Agent Teams

Agent teams 最適合用於並行探索能增加真實價值的任務：

  • 研究和審查：多個隊友可以同時調查問題的不同方面
  • 新模組或功能：隊友可以各自擁有一個獨立的部分
  • 使用競爭假設進行除錯：隊友並行測試不同的理論
  • 跨層協調：跨越前端、後端和測試的更改

Agent teams 增加了協調開銷，使用的 token 數量明顯多於單個工作階段。當隊友可以獨立運作時效果最好。

與 Subagents 比較

Subagents Agent teams
Context 自己的 context window；結果回傳給呼叫者 自己的 context window；完全獨立
通訊 僅向主代理回報結果 隊友直接相互傳送訊息
協調 主代理管理所有工作 具有自我協調的共享任務清單
最適合 只有結果重要的專注任務 需要討論和協作的複雜工作
Token 成本 較低：結果彙總回主 context 較高：每個隊友是一個獨立的 Claude 實例

當你需要快速、專注的工作者回報結果時，使用 subagents。當隊友需要分享發現、相互質疑和自我協調時，使用 agent teams。

控制你的 Agent Team

選擇顯示模式

  • In-process（程序內）：所有隊友在主終端內執行。使用 Shift+Down 循環瀏覽。
  • Split panes（分割窗格）：每個隊友取得自己的窗格。需要 tmux 或 iTerm2。

設定中的 teammateMode 選項："auto"（預設）、"in-process" 或 "tmux"。

claude --teammate-mode in-process

指定隊友和模型

Use Sonnet for each teammate.

要求隊友的計畫批准

直接與隊友交談

  • In-process 模式：使用 Shift+Down 循環瀏覽隊友，按 Ctrl+T 切換任務清單
  • Split-pane 模式：點選隊友的窗格

分配和認領任務

共享任務清單協調整個團隊的工作。任務有三種狀態：待處理（pending）、進行中（in-progress）和已完成（completed）。

Ask the researcher teammate to shut down

清理團隊

Clean up the team

始終使用負責人進行清理。隊友不應該執行清理。

使用 Hooks 強制品質門

  • TeammateIdle：當隊友即將閒置時執行。以代碼 2 退出以傳送回饋並保持隊友工作。
  • TaskCompleted：當任務被標記為完成時執行。以代碼 2 退出以防止完成並傳送回饋。

Agent Teams 如何工作

架構

元件 角色
Team lead（團隊負責人） 建立團隊、產生隊友並協調工作的主 Claude Code 工作階段
Teammates（隊友） 各自處理分配任務的獨立 Claude Code 實例
Task list（任務清單） 隊友認領和完成的共享工作項清單
Mailbox（信箱） 代理之間通訊的訊息系統

團隊和任務儲存在本地：

  • Team config：~/.claude/teams/{team-name}/config.json
  • Task list：~/.claude/tasks/{team-name}/

權限

隊友從負責人的權限設定開始。如果負責人使用 --dangerously-skip-permissions 執行，所有隊友也會如此。

Context 和通訊

每個隊友都有自己的 context window。產生時，隊友載入與常規工作階段相同的專案 context：CLAUDE.md、MCP 伺服器和技能。

  • 自動訊息傳遞：當隊友傳送訊息時，它們會自動傳遞給收件人
  • 閒置通知：當隊友完成並停止時，自動通知負責人
  • 共享任務清單：所有代理都可以看到任務狀態

Token 使用

Agent teams 使用的 token 數量明顯多於單個工作階段。每個隊友都有自己的 context window。對於研究、審查和新功能工作，額外的 token 通常是值得的。

用例範例

執行並行程式碼審查

Create an agent team to review PR #142. Spawn three reviewers:
- One focused on security implications
- One checking performance impact
- One validating test coverage
Have them each review and report findings.

使用競爭假設進行調查

Users report the app exits after one message instead of staying connected.
Spawn 5 agent teammates to investigate different hypotheses. Have them talk to
each other to try to disprove each other's theories, like a scientific
debate. Update the findings doc with whatever consensus emerges.

最佳實踐

  • 給隊友足夠的上下文：在產生提示中包含特定於任務的詳細資訊
  • 選擇適當的團隊規模：對大多數工作流，從 3-5 個隊友開始
  • 適當調整任務大小：自包含的單位，產生清晰的可交付成果
  • 等待隊友完成：告訴負責人在繼續前等待隊友
  • 從研究和審查開始：如果你是 agent teams 的新手
  • 避免檔案衝突：分解工作使每個隊友擁有不同的檔案集
  • 監控和引導：不要讓團隊無人值守執行太長時間

疑難排解

  • 隊友未出現：在 in-process 模式中按 Shift+Down 循環瀏覽。檢查 tmux 是否已安裝。
  • 過多權限提示：在權限設定中預批准常見操作。
  • 隊友在錯誤後停止：使用 Shift+Down 或點選窗格檢查輸出，然後給予額外指示。
  • 負責人在工作完成前關閉：告訴它繼續或等待隊友完成。
  • 孤立的 tmux 工作階段：tmux ls 列出，tmux kill-session -t <session-name> 清理。

限制

  • In-process 隊友沒有工作階段恢復：/resume 和 /rewind 不會恢復 in-process 隊友
  • 任務狀態可能滯後：隊友有時無法將任務標記為已完成
  • 關閉可能很慢：隊友在關閉前完成目前的請求或工具呼叫
  • 每個工作階段一個團隊：負責人一次只能管理一個團隊
  • 沒有巢狀團隊：隊友無法產生自己的團隊
  • 負責人是固定的：無法將隊友提升為負責人或轉移領導權
  • 權限在產生時設定：所有隊友從負責人的權限模式開始
  • 分割窗格需要 tmux 或 iTerm2：VS Code 整合終端、Windows Terminal 或 Ghostty 不支援分割窗格模式

提示：CLAUDE.md 正常工作：隊友從工作目錄讀取 CLAUDE.md 檔案。
